已订阅问题 webhooks 的 Sentry 集成可以接收有关问题创建和状态更改的信息。

## Sentry-Hook-Resource 标头

`'Sentry-Hook-Resource': 'issue'`

## 属性

### action

- 类型：字符串
- 描述：可以是 `created`、`resolved`、`assigned`、`archived` 或 `unresolved`

### data['issue']

- 类型：对象
- 描述：问题

### data['issue']['url']

- 类型：字符串
- 描述：问题的 api url

### data['issue']['web_url']

- 类型：字符串
- 描述：问题的 web url

### data['issue']['project_url']

- 类型：字符串
- 描述：问题所属项目的 api url

## 有效负载

有效负载包括有关问题的附加信息。需要注意的一些重要字段是：

### status

一些问题 webhooks 是在问题状态更改时触发的。问题状态可以是 `resolved`、`unresolved` 或 `ignored`（已归档）。

### substatus

状态可能意味着多种事情，因此子状态提供了有关状态的更多详细信息。例如，问题状态可能更改为 `unresolved`，因为它现在是 `escalating` 或因为已解决的问题已 `regressed`。可能的子状态有：`archived_until_escalating`、`archived_until_condition_met`、`archived_forever`、`escalating`、`ongoing`、`regressed` 和 `new`。

### statusDetails

`statusDetails` 包括有关问题的 `resolved` 和 `archived` 状态更改的更详细信息。这些详细信息可以包括：
* `inRelease`：解决所在的版本（例如 `latest`）
* `inNextRelease`：如果解决将在下一个版本中，则为 `True`
* `inCommit`：包括有关解决提交的信息（如 `repository` 和 `commit`）
* `ignoreCount`：已归档问题在被升级之前的最大出现次数
* `ignoreWindow`：与 `ignoreCount` 一起使用，表示 `ignoreCount` 出现将被忽略的分钟数
* `ignoreUserCount`：已归档问题在被升级之前受影响的最大用户数
* `ignoreUserWindow`：与 `ignoreUserCount` 一起使用，表示 `ignoreUserCount` 受影响用户将被忽略的分钟数
* `ignoreDuration`：已归档问题在被升级之前应被忽略的持续时间（以分钟为单位）。受影响用户数量或出现次数的增加不会在此持续时间过去之前升级，除非有峰值。要了解有关其工作原理的更多信息，请参见[升级问题算法](/product/issues/states-triage/escalating-issues/)。

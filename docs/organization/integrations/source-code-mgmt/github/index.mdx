有两种类型的 GitHub 集成，因此请确保您选择了正确的一种：
- [GitHub](#安装-github) - 适用于个人 GitHub 用户/组织（以及大多数 GitHub Enterprise Cloud 客户），只要它们托管在 `github.com` 上。请遵循"[安装 GitHub](#安装-github)"说明，**请从 Sentry 开始安装**。

- [GitHub Enterprise](#安装-github-enterprise) - 适用于**无法从 `github.com` 访问**的 GitHub Enterprise 实例。通常这些将具有自定义域，例如：
  - `https://github.example.com`
  - `https://ghe.example.com/`

  对于这些账户，请遵循"[安装 GitHub Enterprise](#安装-github-enterprise)"说明。

## 安装 GitHub


1. 在 Sentry 中，导航到**设置 > 集成 > [GitHub](https://sentry.io/orgredirect/organizations/:orgslug/settings/integrations/github)**。

   ![安装 GitHub 集成](./img/github-install-page.png)

2. 如果您已安装旧版 GitHub 集成，请使用 GitHub 旁边的按钮**升级**。如果您没有安装旧版 GitHub 集成，请使用按钮**安装**。

3. 在结果模态框中，点击"添加安装"。

4. 应该会弹出一个 GitHub 安装窗口。点击"安装"。
5. 选择 Sentry 应该有权访问的存储库（或选择所有存储库）。
6. 然后您应该被重定向回 Sentry 集成页面。
7. 在 Sentry 中的新 GitHub 实例上，点击"配置"。
8. 默认情况下将添加您的所有存储库，您只需刷新屏幕即可。要仅添加选定的存储库，请从下拉列表中选择它们。_请确保您在前面的步骤中已在 GitHub 中授予 Sentry 对这些存储库的访问权限。_

   ![添加存储库](./img/github-global-add-repo.png)

9. 您现在可以[配置](#配置)集成。

GitHub 集成适用于您 Sentry 组织下的所有项目。您可以将多个 GitHub 组织连接到一个 Sentry 组织，但您**不能**将单个 GitHub 组织连接到多个 Sentry 组织。

虽然您可以从 [GitHub](https://github.com/apps/sentry-io) 安装 GitHub 集成，但我们建议从 Sentry 安装以获得更简化的过程。

### GitHub 权限

以下权限是 GitHub 集成正常工作所必需的。您必须完全选择加入这些权限才能使用该应用。除非另有说明，否则列出的所有权限都是存储库级别的权限。

| 权限类别 | 访问级别 | Sentry 需要它的原因 |
|--------------------|---------------|-----------------------------------------------------------------------------------------------------------------------------------|
| 管理 | 只读 | 尊重分支保护规则、确定默认分支并增强自动化。 |
| 检查 | 读取和写入 | 在 PR 或提交上显示详细的通过/失败结果（例如测试覆盖率、发布运行状况）作为 GitHub 检查。 |
| 提交状态 | 读取和写入 | 向 GitHub 中的提交发布状态检查（例如测试分析数据）并与分支保护规则集成。 |
| 内容 | 只读 | 获取用于可疑提交、跟踪发布中的文件的源文件、提交和责备数据。 |
| 问题 | 读取和写入 | 从 Sentry 事件创建 GitHub 问题并更新链接的问题。 |
| <span>成员<br/><em>_[组织级别]_</em></span> | 只读 | 将组织用户映射到 GitHub 身份以改进问题归因和协作。 |
| 元数据 | 只读 | 识别存储库结构、名称、默认分支和可见性。 |
| 拉取请求 | 读取和写入 | 在拉取请求上写评论（1）由拉取请求引起的问题和（2）突出显示代码差异中的现有问题。|
| Webhooks | 读取和写入 | 订阅推送事件、PR 和问题的实时更新。

偶尔，Sentry 会在引入新功能时请求对您的 GitHub 帐户的额外权限。拒绝或忽略更新请求不会影响您当前的 Sentry 使用，但可能会阻止访问未来功能。

### 安装您的 GitHub 应用

1. 在 Sentry 中，导航到组织设置 > **集成**。
2. 在 GitHub Enterprise 旁边，点击"安装"。

   ![安装 GitHub Enterprise 集成](./img/github-e-install.png)

3. 点击"添加安装"。

   ![添加安装](./img/github-e-add-installation.png)

4. 使用来自您的 GitHub 应用配置页面的信息填写以下表单。安装 URL 必须能够通过互联网解析。

   ![GitHub Enterprise 配置表单](./img/github-e-form.png)

   您需要在您的 GitHub 应用配置页面上生成一个私钥，并将整个内容粘贴到 **GitHub 应用私钥** 字段中。

   ![生成私钥](./img/github-e-generate-private-key.png)

   例如，在终端中：

   ```
   cat <您的私钥文件> | pbcopy
   ```

5. 点击"配置"，然后会弹出一个 GitHub 安装窗口。选择 Sentry 应该有权访问的存储库（或选择所有存储库）并点击"安装"。

   ![安装 GitHub Enterprise 确认模态框](./img/github-e-repo-access.png)

6. 然后您将被重定向回 Sentry。在您的新 GitHub Enterprise 实例上，点击"配置"。

   ![配置 GitHub enterprise](./img/github-e-configure.png)

7. 添加您想要收集提交数据的任何存储库。_请确保您在前面的步骤中已在 GitHub 中授予 Sentry 对这些存储库的访问权限。_
   ![添加 GitHub Enterprise 存储库](./img/github-e-add-repo.png)

GitHub Enterprise 现在应该已为您 Sentry 组织下的所有项目启用。

## 配置

使用 GitHub 来[跟踪提交](#提交跟踪)、识别[可疑提交](#可疑提交和建议的受让人)、[通过提交或 PR 解决](#通过提交或拉取请求解决)和[管理问题](#问题管理)。

### 提交跟踪

提交跟踪允许您专注于有问题的提交。了解有关[提交跟踪](/product/releases/setup/release-automation/)的更多信息。

### 可疑提交和建议的受让人

一旦您设置了提交跟踪，您将能够看到问题堆栈跟踪中找到的文件的最最近更改，这些更改来自可疑提交。

对于堆栈跟踪中的文件与发送到 Sentry 的提交中包含的文件匹配的问题，您将看到可疑提交，并带有指向提交本身的链接。

![可疑提交](./img/github-global-suspect-commits.png)

您还将看到可疑提交的作者被列为该问题的建议受让人。要将问题分配给建议的受让人，请点击他们的图标。

### 问题管理

问题跟踪允许您从 Sentry 内创建 GitHub 问题，并将
Sentry 问题链接到现有的 GitHub 问题。

<Alert title="注意" level="info">
  <p>
    手动问题管理适用于团队、商业或
    企业计划的组织。
  </p>
  <p>
    自动问题管理适用于商业或企业计划的组织。
  </p>
</Alert>

问题管理可以通过两种方式配置 - 自动或手动。

#### 自动

要自动配置问题管理，请创建[**问题警报**](/product/alerts-notifications/issue-alerts/)。在选择[**操作**](/product/alerts/create-alerts/issue-alert-config/#then-conditions-actions)时，选择**创建新的 GitHub 问题**。

![创建 GitHub 问题警报规则操作](./img/github-ticket-rule.png)

当警报被触发时将自动创建 GitHub 问题。

#### 手动

要手动配置问题管理，一旦您导航到特定的 Sentry 问题，您将在右侧面板中找到**链接问题**部分。

![从 Sentry 问题链接 GitHub 问题](./img/github-global-link-issue.png)

在这里，您将能够创建或链接 GitHub 问题。

![创建 GitHub 问题表单](./img/github-create-issue.png)

### 通过提交或拉取请求解决

一旦您正在发送提交数据，您就可以开始通过在提交消息中包含 `fixes <SENTRY-SHORT-ID>` 来解决问题了。例如，提交消息可能如下所示：

```
防止用户上的空查询

修复 MYAPP-317
```

您还可以通过在标题或描述中包含 `fixes <SENTRY-SHORT-ID>` 来使用拉取请求解决问题。

当 Sentry 看到这一点时，我们将自动用对提交或拉取请求的引用来注释匹配的问题，并且稍后，当该提交或拉取请求是发布的一部分时，我们将问题标记为已解决。提交必须与发布相关联。否则，如果提交被压缩，Sentry 将不知道何时发布了提交，问题可能永远不会被标记为回归。使用 GitHub，我们建议在我们将要合并到您的主分支的拉取请求的正文中使用 `fixes <SENTRY-SHORT-ID>`。

### 在拉取请求上获取 Sentry 评论

Sentry 可以对已合并和开放的拉取请求发表评论。这些功能在您的 GitHub 集成设置完成后会自动启用。要禁用它，请转到**设置 > 集成 > GitHub** 并将其关闭。

![如何禁用 Sentry 对 GitHub 拉取请求的评论](./img/github-org-settings.png)

#### 已合并的拉取请求评论

如果最近合并的拉取请求被怀疑引起了问题，Sentry 将对其进行评论以显示问题，如下例所示。

![Sentry 对 GitHub 中可疑拉取请求的评论](./img/sentry-github-pr-comment.png)

Sentry 只会对小于两周且与可疑提交相关联的拉取请求发表评论。该评论将不断更新，最多包含拉取请求被怀疑引起的五个问题。

#### 开放的拉取请求评论

当打开拉取请求时，Sentry 将从 PR 差异中解析拉取请求中修改的文件名和函数名，以查找与您的拉取请求相关联的最近的未处理、未解决的问题。

![Sentry 对 GitHub 中开放拉取请求的评论](./img/sentry-github-open-pr-comment.png)

Sentry 将使用每个文件最多五个问题对拉取请求发表评论，这些问题在过去 90 天内首次出现，并在过去 14 天内最后一次出现。

此功能需要[代码映射](/product/issues/suspect-commits/#2-set-up-code-mappings)，目前仅支持 Python、JavaScript/TypeScript、PHP 和 Ruby 文件。如果您使用不同的语言，[请在此 GitHub 工单上告知我们](https://github.com/getsentry/sentry/issues/69824)。对于 JavaScript/TypeScript，请确保您已通过[设置源映射](/platforms/javascript/sourcemaps/)取消压缩了代码。

### 缺失成员检测

如果有用户向链接到 Sentry 的 GitHub 存储库提交代码，但他们不是您组织的成员，Sentry 会将其检测为缺失成员。每月一次，Sentry 会向组织所有者和管理员发送电子邮件，提醒他们邀请这些用户加入他们的组织。Sentry 还会在**设置 > 成员**页面中显示邀请缺失成员的横幅。

![GitHub 缺失成员邀请横幅](./img/github-missing-members-banner.png)

该横幅显示尚未加入组织的 5 个最活跃的贡献者。要查看所有缺失成员（最多 50 个），请点击横幅中的"查看全部"。要暂停横幅，请点击省略号，然后选择"隐藏缺失成员"。

点击邀请横幅上的"查看全部"将显示邀请模态框。从这里，您可以选择多个用户并在邀请他们之前分配角色和团队。

![GitHub 缺失成员邀请模态框](./img/github-missing-members-modal.png)

默认情况下，此功能在您的 GitHub 集成设置完成后会自动启用。要禁用它，请转到**设置 > 集成 > GitHub** 并将其关闭。

![如何禁用缺失成员检测](./img/github-org-settings.png)

### 堆栈跟踪链接

<Alert>

Sentry 将自动尝试为安装了 GitHub 集成的组织在 C#、Clojure、Go、Groovy、Java、JavaScript、PHP、Python、Ruby 和 Scala 上设置代码映射。如果您的项目使用不同的 SDK，您可以手动添加代码映射。

</Alert>

堆栈跟踪链接将您从 Sentry 堆栈跟踪中的文件带到源代码中的同一文件。如果您在 Sentry 中设置了提交跟踪，我们可以带您到源代码的确切版本（使用与事件关联的提交）。否则，我们将链接到源代码的当前状态（使用默认**分支**）。

1. 导航到**设置 > 集成 > GitHub > 配置**。

1. 点击您的 GitHub 实例旁边的"配置"按钮。

1. 点击**代码映射**选项卡。

1. 为您想要启用堆栈跟踪链接的每个项目设置代码映射。要创建新的代码映射，请点击**添加映射**。

1. 填写表单，然后点击**保存更改**。每个表单字段描述如下：

   - **项目**（必需）：这是 Sentry 项目。
   - **存储库**（必需）：这是与上述 Sentry 项目关联的 GitHub 存储库。如果您每个 Sentry 项目使用多个 GitHub 存储库，您将需要多个代码映射。
   - **分支**（必需）：这是我们在没有设置提交跟踪时回退到的代码的默认分支。
   - **堆栈跟踪根**和**源代码根**（可选）：有关确定这些值的信息，请参见下文。

#### 堆栈跟踪根和源代码根

<Alert>

以下信息仅对使用传统文件路径的平台有效。具有包名称的平台需要额外的步骤。对于**Java**，请在[源上下文页面](/platforms/java/source-context#setting-up-code-mappings)中阅读更多信息。

</Alert>

首先，导航到您希望映射的堆栈跟踪。找到一个**应用内**框架，该框架在框架右侧用气泡表示。文件名将显示为框架标题左侧的第一个文本。在此示例中，它是 `src/main.py`。

<Include name="common-imgs/code-mappings-stacktrace" />

对于某些原生平台，堆栈跟踪看起来会有所不同。在这种情况下，您可以通过将鼠标悬停在文件名上来找到绝对路径。

![突出显示在 UI 中查找原生堆栈跟踪文件路径的位置](./img/code-mappings-stacktrace-native.png)

如果您不确定，您可以通过点击事件标题中的 `{}` 按钮来查看事件 JSON。在框架的 `filename` 或 `abs_path` 中找到文本。

<Include name="common-imgs/code-mappings-event-json" />

接下来，在您的源代码提供商（例如 GitHub）中找到堆栈跟踪中看到的文件。在此示例中，路径是 `flask/src/main.py`（`empower` 被忽略，因为它是存储库的名称）。

<Include name="common-imgs/code-mappings-source-file" />

将堆栈跟踪中的文件路径与源代码中的路径进行比较。在此示例中，堆栈跟踪中的 `src/` 文件夹与源代码中的 `flask/src` 文件夹匹配。使用该信息，将**堆栈跟踪根**设置为 `src/`，将**源代码根**设置为 `flask/src/`。这告诉 Sentry 在搜索源代码时将所有以 `src/` 开头的文件路径替换为 `flask/src/`。

为了获得最佳结果，我们建议尽可能始终为**堆栈跟踪根**提供非空值。

## 故障排除

如果您在添加 GitHub 存储库时遇到问题：

- 确保您是 GitHub 组织的所有者。
- [断开连接](https://sentry.io/account/settings/identities/)，然后[重新连接](https://sentry.io/account/settings/social/associate/github/)您的 GitHub 身份。

如果您在设置 Sentry 与您的本地集成时遇到问题，请验证以下内容：

- 提供的安装 URL 是完全限定域名 (FQDN)，可在互联网上解析。
- Sentry 用于发出出站请求的 IP 地址是[允许的](/security-legal-pii/security/ip-ranges/)。
- Sentry 对您的安装 URL 的访问不受路径限制。

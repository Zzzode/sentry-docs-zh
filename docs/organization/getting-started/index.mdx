在本指南中，我们将提供设置 [Sentry 组织账户](https://sentry.io) 的推荐清单，以便您开始使用 Sentry 错误监控。

如果您更喜欢 TL;DR，请下载我们的 ["管理员快速参考指南"](/pdfs/Admin-quick-reference-guide.pdf)

我们理解你们中有些人正在运营企业组织，而有些人只是出于爱好编码。下面，我们根据您的情况链接了您应该从哪里开始。

- 您正在为大型组织配置 Sentry？从 [第 1 步](#1-启用单点登录访问) 开始
- 您正在与一个小团队一起开发应用程序？从 [第 2 步](#2-设置团队) 开始
- 您是爱好者或独自开发应用程序？从 [第 3 步](#3-配置集成) 开始

当然，即使您是一个人团队，也欢迎您完成所有步骤。

## 1. 启用单点登录访问

[单点登录 (SSO)](/organization/authentication/sso/) 允许您的团队快速登录，简化成员账户的入职/离职流程，并通过安全凭据加强登录安全性。Sentry 提供开箱即用的集成配置，用于集成 SSO 提供商，如 [Okta](/organization/authentication/sso/#okta) 和 [Azure Active Directory](/organization/authentication/sso/#azure-active-directory) (SAML) 或 [Google](/organization/authentication/sso/#google-business-app) 和 [GitHub](/organization/authentication/sso/#github-organizations) (Oauth)。此外，我们还为任何其他 [SAML2 身份提供商](/organization/authentication/sso/saml2/) 提供通用配置选项。

Sentry 还支持跨域身份管理系统 (SCIM) 规范的子集，用于 [Okta](/organization/authentication/sso/okta-sso/okta-scim/) 和 [Azure AD](/organization/authentication/sso/azure-sso/#scim-integration)。

## 2. 设置团队

现在 SSO 已配置且成员已创建他们的账户，将他们添加到他们的 Sentry 团队中。团队与您的 [Sentry 项目](#4-创建项目) 相关联，其成员除了成为问题受让人外，还会收到问题通知。我们建议创建与您内部团队结构一致的 Sentry 团队（例如，_#前端_、_#运维_、_#SDK_ 等）。要添加新团队，请转到 **设置 > 团队** 并点击"创建团队"。

![创建新团队](./img/team-settings.png)

点击团队名称打开团队设置并管理其关联成员和项目。
在"成员"选项卡下，通过点击"添加成员 > [成员名称]"将现有成员添加到您的团队。

您还可以通过点击"添加成员 > 邀请成员"邀请多个新的（外部）成员加入您的组织和团队。

![邀请成员](./img/invite-members.png)

了解更多关于不同用户角色的信息，请访问 [组织管理](/organization/membership/)。

## 3. 配置集成

Sentry 通过提供与广泛使用的应用程序和服务的开箱即用集成来集成到您现有的工作流程中。要启用和配置集成，请转到 **设置 > 集成**。有几种可用的集成类型，但我们建议您立即设置以下集成：

- [警报](#31-警报通知)
- [源代码管理](#32-源代码管理)
- [问题跟踪](#33-问题跟踪)

### 警报通知

默认情况下，Sentry 将通过电子邮件和 [Slack](/organization/integrations/notification-incidents/slack/)（一旦您启用集成）通知您应用程序中的错误。您还可以在 Slack 中启用团队通知，以便您组织中的合适人员始终收到他们需要的警报。除了 Slack，Sentry 还提供其他几种 [通知和事件集成](/organization/integrations/notification-incidents/) 来帮助您分类问题，包括 [Discord](/organization/integrations/notification-incidents/discord/)、[Microsoft Teams](/organization/integrations/notification-incidents/msteams/) 和 [PagerDuty](/organization/integrations/notification-incidents/pagerduty/)。

了解更多关于使用集成设置警报的信息，请访问 [使用集成进行警报路由](/product/alerts/create-alerts/routing-alerts/)。

### 源代码管理

当您启用与源代码管理 (SCM) 提供商的集成时，Sentry 将分析您的提交数据以：

1. 识别 **可疑提交**，这些提交可能引入了错误。
1. **建议受让人** 谁可以最好地解决错误，基于可疑提交和您的 [代码所有者文件](/product/issues/issue-owners/#code-owners)。
1. 将问题标记为 **通过提交或 PR 解决**，以停止关于过去版本中类似错误的通知，并通知您回归。
1. 将 Sentry 问题链接到您集成的 [问题跟踪](#33-问题跟踪) 解决方案中的新问题或现有问题。

   ![识别可疑提交并查看问题的建议受让人。](./img/issue-integrations-01.png)

   ![将 Sentry 问题链接到问题跟踪解决方案中的新问题或现有问题。](./img/issue-integrations-02.png)

Sentry 内置的 SCM 集成包括：

- [Azure DevOps](/organization/integrations/source-code-mgmt/azure-devops/)
- [GitHub](/organization/integrations/source-code-mgmt/github/)
- [GitLab](/organization/integrations/source-code-mgmt/gitlab/)
- [Bitbucket](/organization/integrations/source-code-mgmt/bitbucket/)

如果您使用不同的 SCM 提供商或不想让 Sentry 连接到您的存储库，请查看我们的 [发布文档](/product/releases/associate-commits/) 了解如何仍然将提交与您的发布关联。

### 问题跟踪

启用与问题跟踪解决方案的集成允许您从 [sentry.io](https://sentry.io) 中的 **问题详情** 页面创建新问题，或链接到现有问题。GitHub、GitLab 和 Bitbucket 问题与其各自的 SCM 集成相关联。Sentry 还与 [Azure DevOps](/organization/integrations/source-code-mgmt/azure-devops/)、[Shortcut](/organization/integrations/issue-tracking/shortcut/)、[Jira](/organization/integrations/issue-tracking/jira/) 等集成。

有关所有受支持集成的列表，请查看我们的 [完整集成文档](/organization/integrations/)。

您可以在创建路由到 [Azure DevOps](/organization/integrations/source-code-mgmt/azure-devops/#automatically) 和 [Jira](/organization/integrations/issue-tracking/jira/#automatically) 的警报时设置自动化问题管理。将代表您为新 Sentry 问题创建外部问题，这些问题将使用 Azure DevOps 或 Jira 工作区中配置的字段。对于其他问题跟踪解决方案，您可以手动链接 Sentry 问题。

<Alert>

自动问题管理仅适用于您的组织在业务或试用计划上。

</Alert>

## 4. 创建项目

要开始用 Sentry 监控应用程序中的错误，您需要使用 DSN 密钥初始化 SDK。要获取密钥，请通过转到 **项目** 并点击"创建项目"来添加新的 Sentry 项目。为项目命名并分配负责的 [团队（或团队）](#2-设置团队)。然后，在 **[项目] > 设置 > SDK 设置 > 客户端密钥 (DSN)** 中检索密钥。

![检索您的项目 DSN 密钥。](./img/project-dsn.png)

一旦 SDK 初始化，无论代码部署和运行在哪里，其中发生的任何错误都将与该特定项目关联。

如果您没有设置任何团队与您的项目关联，因为您跳到了第三步，请不要担心。即使在我们的免费开发者计划中，Sentry 也会根据您组织的名称自动为您设置一个团队，并将您添加到其中。

### 项目中有什么？

_项目_ 是一个逻辑实体，将应用程序（或其一部分）中的错误连接到分配给该项目的团队成员。项目设置确定：

- 哪些错误应摄取到您的 Sentry 账户中，哪些应通过 [入站过滤器](/pricing/quotas/#inbound-data-filters) 删除
- 通知谁、关于什么错误以及如何通知，使用 [警报规则](/product/alerts/alert-types/)
- 哪些错误应使用 [所有权规则](/product/issues/ownership-rules/) 自动分配给哪个成员或团队
- 用于微调 [事件分组](/concepts/data-management/event-grouping/) 为问题的自定义规则

此外，当您转到 [sentry.io](https://sentry.io) 中的 **问题** 和 **发现** 页面时，屏幕顶部的过滤器默认优先考虑您是成员的项目。这样，开发人员正在查看与其工作相关的信息，并且他们看到可能需要处理的问题。

![带有项目下拉菜单打开的问题页面。](./img/issues-project-dropdown.png)

### 我应该创建多少个项目？

理论上，您可以将所有错误放入一个项目中，因为这在 [sentry.io](https://sentry.io) 中不受限制。但是，设置多个项目以更精细的粒度反映您的应用程序有助于使错误更明显和可操作，这对您团队的生产力有很大影响。

以下是一些需要考虑的事项：

- 如果您的应用程序的源代码在 **多个存储库** 中管理，请为 **每个存储库创建一个单独的项目**。
- 如果您的应用程序由 **几个微服务** 组成，请相应地 **将它们拆分为项目**。
- 如果您有 **单体代码库**，请为 **后端和前端设置单独的项目**。
- 为 **每种语言提供自己的项目**。例如，如果您的后端代码包含 NodeJS 和 Java 组件，请将它们分成两个不同的项目。

## 5. 定义环境

根据您的开发生命周期，您的应用程序可能部署并运行在多个环境中 — _QA_、_暂存_、_生产_ 等。在 Sentry SDK 中配置环境将为每个错误事件添加标签，并帮助您将事件与它们发生的环境关联起来。您可以通过创建 `SENTRY_ENVIRONMENT` 环境变量或通过显式在 SDK 配置中定义它来实现。了解更多信息，请访问 [环境](/platform-redirect/?next=/configuration/environments/)。

![带有环境下拉菜单打开的问题页面。](./img/environment-dropdown.png)

在 SDK 中定义环境将允许您：

- 创建特定于环境的警报规则。
- 根据环境过滤 **问题** 页面。
- 通过 **发现** 中的查询，基于环境获得比较洞察。

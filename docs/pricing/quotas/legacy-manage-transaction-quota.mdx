<Alert>
此信息仅适用于在 **2024年6月11日之前** 激活且之后未升级的付费计划。

如果您使用的是在2024年6月11日或之后激活的免费或付费计划，请参阅[当前关于管理跨度配额的页面](/pricing/quotas/manage-transaction-quota/)。
</Alert>

在大多数情况下，将所有事务和带有性能分析的事务事件发送到 Sentry 可能会产生比您需要的更多性能数据，并且可能会过快地用完您的性能单元。Sentry 提供了控制被监控的事务和带有性能分析的事务的 _类型_ 和 _数量_ 的工具。这使您能够拥有可操作且有意义的性能数据，并管理您的配额。

应用适当的 SDK 配置是一个迭代且持续的过程，但这些技巧将向您展示如何最好地使用 Sentry 的工具来充分利用您的事务和带有性能分析的事务事件。这些技巧从最简单或最省时的到最具挑战性或可能最耗时的进行排序。前两个选项是您在 [sentry.io](https://sentry.io) 中可以做的事情，而其余的则是您在 SDK 中要做的事情。

## 开始之前：检查您的配额使用情况

<Alert title="注意">

"使用情况统计"选项卡仅在您使用团队、商业或企业计划时可见。

</Alert>

您可以在 **统计** 的"使用情况统计"选项卡中查看您的聚合事件。这些信息将帮助您回答关于传入事件分解或哪些项目正在消耗您的配额的关键问题。这些问题的答案可以帮助您确定需要在何处对 SDK 过滤器和配置进行进一步微调。

此页面对您组织的所有成员都可访问，因此您 Sentry 组织中的所有者可以与直接负责给定项目的开发人员共享此页面。此外，您可以返回此页面检查您所做的更改是否产生了预期效果。

### 如何查看传入事件的分解？

[使用情况统计](/product/stats/#usage-stats)选项卡显示整个组织最多90天内 Sentry 接收的数据总量的详细信息。该页面将事件（按项目）分为三类：_已接受_、_已丢弃_ 或 _已过滤_。只有已接受的事件会影响您的配额：

![使用情况统计页面显示事务](./img/usage-stats-transactions.png)

### 我最忙的项目是什么？

**统计** 的"使用情况统计"选项卡中的"项目"表按项目分解您的事件，以便您可以看到哪些项目正在消耗您的配额。点击表中项目名称旁边的设置图标将打开项目设置，您可以在其中管理其[入站过滤器](#2-inbound-data-filters)。

### 哪些事务正在消耗我的配额？

<Alert>

只有在您的组织使用商业计划时，您才能更新 **发现** 中结果表的列。

</Alert>

您可以在 **发现** 中设置[查询](<https://sentry.io/orgredirect/organizations/:orgslug/discover/results/?field=title&field=project&field=count()&name=All+Events&query=event.type%3Atransaction&sort=-count&statsPeriod=14d&yAxis=count()>)，以查看您收到最多的事务或带有性能分析的事务，以及您是否想要调整它们。构建查询时，搜索 `event.type:transaction`，然后将列设置为 `title`、`project` 和 `count`，如下所示：

![最繁忙的项目](./img/manage-trans-quota-01.png)

应用更改后，按"COUNT()"列对"结果"表进行排序，以显示您最繁忙的事务：

![最繁忙的问题](./img/manage-trans-quota-02.png)

您可以使用 `has:profile.id` 查找带有性能分析的事务。将 `profile.id` 列添加到查询中将为您提供指向性能分析的链接。

## 调整配额

<Alert>

只有您 Sentry 组织的账单或所有者成员才能更新配额。

</Alert>

一旦您的性能单元量接近或已超过配额，具有"所有者"组织权限级别的队友将收到[通知](/product/alerts/notifications/#quota-notifications)电子邮件。然后您可以选择[增加](#increasing-quotas)或[减少](#decreasing-quotas)您的配额。

### 增加配额

如果您因为超出配额而丢弃了您想要保留的事件，请通过在计费期间的任何时间增加您的[预留](#reserved-volume)或[按需](#on-demand-budget)配额来添加到您的配额。

当您超出配额阈值时，服务器将响应429 HTTP状态码，该状态码向SDK和客户端传达停止发送事件的信号。此状态码附带一个`Retry-After`标头，指示此速率限制处于活动状态的时间。但是，客户端不应重试事件，而应丢弃事件直到速率限制到期，以防止队列积压。请注意，由于事件摄取和速率限制是异步发生的，429 HTTP状态码总是略有延迟。

要增加您的配额，请转到 **设置 > 订阅** 并点击"管理订阅"按钮以访问您的订阅选项。当您增加配额时，更改会立即生效。

如果您使用的是开发者计划并希望增加配额，则需要升级到团队或商业计划。在这些计划中，您可以预付更多性能单元量并按需购买容量。在我们的[定价页面](https://sentry.io/pricing/)上了解 Sentry 的计划。

<Alert>

如果您在计费期间从团队升级到商业，您的按需定价会追溯更改。

</Alert>

### 预留容量

如果您需要的事件数量稳步增加，您可能需要增加预留容量或量。预留容量比按需容量便宜，因为您需要预付费用。它还允许您提前选择您想要可用的性能单元数量，而不仅仅是设置任意的按需预算。在我们的定价文档中了解更多关于[预留容量](/pricing/#prepaid-reserved-capacity)的信息。

如果您认为您对更多事件的需求是暂时的，则不应增加预留容量，因为[减少预留容量与您的计费周期相关](#decreasing-quotas)。

### 按需预算

如果您需要临时增加性能单元配额，我们建议您添加或增加按需预算。这非常适合像推出新版本应用程序这样的情况，您预计当月会有更多事件。要添加按需容量或量，您可以在共享或按类别（错误、性能单元、重放和附件）的基础上输入每月最大预算。在我们的定价文档中了解更多关于[按需容量](/pricing/#on-demand-capacity)的信息。

### 减少配额

计划降级和预留容量减少在您的计费期结束时处理，剩余容量不能退款。例如，如果您的月度计费期从每月5日开始，而您在6月20日减少了预留容量，则此更改将在7月4日处理。从7月5日开始的计费期将反映您的新预留容量。

<Alert level="warning">

如果您有年度计费期，计划降级和预留容量减少将在您的**下一个计费年度**开始时生效。

</Alert>

对按需容量的更改通常会立即生效，并保证在24小时内生效。但是，您不能将按需预算减少到低于您在当前期间已消耗的金额。

要减少您的配额，请转到 **设置 > 订阅** 并点击"管理订阅"。当您到达"查看并确认"步骤时，将显示这些更改生效的日期：

![订阅更新的查看并确认步骤。](./img/review-subscription.png)

<Alert>

我们强烈建议您在计费期的最后一天**之前**进行订阅更改。根据您的时区，在某些情况下，在计费期的最后一天所做的更改要到下一个计费期才会生效。

</Alert>

## 入站过滤器

虽然 SDK 配置需要更改源代码并取决于您的下一次部署，但服务器端过滤器可以在 **[项目] > 设置 > 入站过滤器** 的"数据过滤器"部分中为每个项目轻松配置。

应用后，您可以使用"入站过滤器"页面顶部提供的图表跟踪过滤的事件（数量和原因）。

![内置入站过滤器](./img/manage-event-stream-03.png)

<Alert>

[sentry.io](https://sentry.io) 中的 **入站数据过滤器** 页面仅指错误，但它也适用于事务、带有性能分析的事务和附件。

</Alert>

Sentry 提供了几种在服务器端过滤所有事件和附件的方法，这些都是在检查潜在速率限制之前应用的。入站过滤器包括：

- 常见浏览器扩展错误
- 来自 `localhost` 的事件
- 已知旧版浏览器错误
- 已知网络爬虫
- 代码的特定发布版本
- 某些 IP 地址

上述前四个选项是您只需打开或关闭的设置。设置其他的解释如下。

在这些检查处理完毕后，未根据这些过滤器丢弃的事件和附件会计入您的配额。它们被接受进入 Sentry，在那里它们被持久化和存储。

### IP 过滤器

如果您有恶意客户端，Sentry 支持阻止 IP 发送数据。导航到 **[项目] > 设置 > 入站过滤器** 以在"IP 地址"字段中添加 IP 地址（或子网）。

### 按发布版本过滤

<Include name="feature-available-for-plan-business.mdx" />

如果您发现导致过多噪音的问题发布版本，Sentry 支持忽略来自该发布版本的所有事件和附件。导航到 **[项目] > 设置 > 入站过滤器**，然后将发布版本添加到"发布版本"字段。

## SDK 过滤：beforeSendTransaction

某些 Sentry SDK 支持 `beforeSendTransaction` 回调方法。实现后，当 SDK 捕获事务事件时，在将其发送到您的 Sentry 账户之前调用此方法。由于它以事务事件对象作为参数接收，您可以使用它来根据事件上可用的自定义逻辑和数据（如 _标签_、_环境_、_发布版本_、_事务名称_ 等）修改事件数据或完全丢弃事件（通过返回 `null`）。请注意，只有事务事件通过 `beforeSendTransaction`。错误和消息事件有一个单独的方法 `beforeSend`，所有 SDK 都支持该方法。在[过滤事件](/platform-redirect/?next=/configuration/filtering/)中了解有关这两种方法的更多信息。

## SDK 配置：跟踪选项

配置 SDK 时，您可以通过设置[跟踪选项](/platform-redirect/?next=/configuration/options/%23tracing-options)来控制发送到 Sentry 的事务数量，包括设置采样率或为采样提供函数。您还可以设置[自定义检测](/platform-redirect/?next=/tracing/instrumentation/custom-instrumentation/)以进行性能监控，以捕获某些类型的事务。

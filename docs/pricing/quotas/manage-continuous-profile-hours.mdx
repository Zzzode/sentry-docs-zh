您可以控制连续性能分析何时处于活动状态，并使用采样来控制您被计费的连续性能分析小时数。请参见下文了解如何操作。

## 开始之前：检查您的使用情况

您可以在 **统计** 页面的"使用情况统计"选项卡中查看您的连续性能分析小时数的聚合情况，以了解传入性能分析的分解情况以及哪些项目正在消耗您的配额。这可能有助于您确定需要在何处进一步微调配置。

## 如何对性能分析进行采样？

连续性能分析仅支持**客户端**采样。要控制客户端采样行为，您可以将 `profile_session_sample_rate` 配置参数设置为 0.0-1.0 之间的值，以控制采样的性能分析会话数量（默认值为 0.0）。

对于连续性能分析，会话在 Sentry SDK 初始化时开始，在服务或进程**终止**时停止。采样仅在进程生命周期内评估一次（在初始化期间）。

例如：如果您在后端服务中使用连续性能分析，其中部署了相同服务的 `N` 个实例（如容器或 pod），`profile_session_sample_rate` 控制这些实例中将启用性能分析的百分比。将 `profile_session_sample_rate` 设置为 0.5 意味着随机选择的 50% 的 `N` 个实例将启用性能分析。此采样率仅在实例重新启动或重新部署时才会重新评估。

## 如何控制性能分析何时处于活动状态？

连续性能分析提供了两种"生命周期"模式，可用于控制性能分析器何时运行。这些生命周期模式通过将 `profile_lifecycle` SDK 配置选项设置为以下两个值之一来控制：

- `manual`（默认）- 这提供了对性能分析何时运行的完全手动控制。您可以使用 `start_profile_session` 和 `stop_profile_session` 函数来精确控制这一点。请注意，性能分析会话必须根据 `profile_session_sample_rate` 的值进行采样，`start_profiler` 才能生效。在未采样的会话上调用 `start_profile_session` 将不起作用。

- `trace` - 这提供了类似于以前基于事务的性能分析产品的行为，其中性能分析器将根据跟踪生命周期自动启动和停止。在此模式下，当至少有一个根跨度/事务处于活动状态时，性能分析器将自动启动，当没有根跨度/事务处于活动状态时，性能分析器将自动停止。此生命周期模式仅在满足以下两个条件时才有效：

    - 性能分析会话已采样（通过 `profile_session_sample_rate`）
    - 跟踪已采样（通过 `traces_sample_rate` 或 `traces_sampler`）

## 如何估算后端连续性能分析的使用情况？

要估算在后端使用连续性能分析时的月度使用情况，计算方法是**每月 730 小时 * 被采样的服务实例数量**。例如，如果服务 A 有 5 个实例，服务 B 有 10 个实例，则总使用量为**(5 + 10) 个实例 x 730 小时 = 每月 10,950 小时**。在此上下文中，"实例"是同一服务的单个部署 - 例如，您可以在 N 个 Kubernetes pod 中部署单个服务，每个 pod 将包含至少一个服务实例。由于每个实例都是独立的，因此它们可以独立进行性能分析。您可以通过不对同一服务的多个实例进行性能分析（减少 `profile_session_sample_rate`）来降低成本 - 如果他们在此示例中仅对每个服务的 1 个实例进行性能分析，则为**2 个实例 * 730 小时 = 每月 1,460 小时**。如果有多个服务，客户还可以选择不对性能不敏感的服务进行性能分析（如吞吐量低的服务）。

## 如何购买连续性能分析？

设置您的按量付费（PAYG）预算，该预算可用于为连续性能分析和其他 Sentry 产品分配支出。有关更多信息，请参阅[定价文档](/pricing/)。

## 如果我已在使用基于事务的性能分析怎么办？

连续性能分析和以前的基于事务的性能分析在同一服务中是**互斥的**，这意味着您**不能**同时使用两者。如果您在 Sentry SDK 配置选项中设置了 `profiles_sample_rate` 或 `profiles_sampler`，则您使用的是基于事务的性能分析。如果您设置了 `profile_session_sample_rate`，则您使用的是连续性能分析。

### 如果我尝试同时激活两者会怎样？

如果您在配置了 `profiles_sample_rate` 或 `profiles_sampler` 的情况下设置 `profile_session_sample_rate`，`profile_session_sample_rate` 将不起作用，基于事务的性能分析将优先。

### 我可以在单独的服务上同时使用基于事务的性能分析和连续性能分析吗？

在**单独**的应用程序或服务中，您**可以**选择混合使用基于事务的性能分析 API 和新的连续性能分析 API。我们建议所有用户迁移到连续性能分析 API，因为基于事务的 API 最终将被弃用，并在主要 SDK 版本中删除。我们将继续支持摄取由旧 SDK 发送的基于事务的性能分析，以保持向后兼容性。

## 如何从基于事务的性能分析迁移到连续性能分析？

- 用 `profile_session_sample_rate` 替换 `profiles_sample_rate` 和 `profiles_sampler` 的使用
- 如果您使用的是 `manual` 性能分析生命周期模式，请添加对 `start_profile_session` 和 `stop_profile_session` 的调用
- 如果您使用的是 `trace` 性能分析生命周期模式，请确保除了 `profile_session_sample_rate` 之外，还适当配置了 `traces_sample_rate` 或 `traces_sampler`

有关更多信息，请参阅详细的[迁移指南](/product/explore/profiling/continuous-ui-profiling-migration-guide/)。

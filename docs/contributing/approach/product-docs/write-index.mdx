这个索引页面编写指南只是一个指南——它不会回答您关于如何编写好的索引页面的所有问题，但它会帮助您有一个好的开始。

<Alert>

虽然我们使用术语"索引页面"来指代Docs中某个功能的主页面，但我们不在实际Docs中使用这个术语。在 [其他指南](#其他指南) 中了解更多。

</Alert>

索引页面：

- 解释功能**是什么**
- 描述功能如何使客户受益，而不会过于偏向营销语言
- 描述功能的机制如何帮助客户实现这些好处
- 提供易于查找的**链接**，指向描述主要功能的子页面和/或如何使用该功能的子页面

索引页面不：

- 描述无法在一句或两句话中描述的**如何操作**；这些信息应该在Docs的子页面上。

## 什么和为什么

- 该功能的电梯推介
  - 功能是什么以及它为客户解决什么问题
- 两到五句话
  - 如果您需要在解释功能提供什么之前解释功能概念，这可以更长（例如，[作为功能/概念的发布与应用程序的发布页面](/product/releases/)）
- 一个像素完美的概览截图，展示功能的实际效果

## 如何：包含的功能 - 页面分解

对于不需要太多解释且具有不同部分的功能使用此方法，如 [使用统计](/product/stats/) - 它非常适合非常包含的功能。

- 描述每个页面部分/元素，重点介绍它如何帮助客户实现此功能的"为什么"。
- 如果需要，添加描述页面部分/元素的标题以使每个文本部分突出。
- 描述此功能如何适应/与应用程序的其他部分交互，如果适用（例如，此页面有一个按钮，可将您带到发现查询）。

## 如何：更广泛的功能 - 好处分解

对于需要更多解释来分解和/或有很多子页面来解释其不同好处的页面使用此方法，如 [洞察](/product/insights/)。

- 描述每个页面部分/元素或子页面如何帮助客户实现此功能的"为什么"。
- 在文本中包含指向相关子页面的链接。
- 适当分组页面部分/元素或子页面。 {/*<!-- 待办事项在这里添加一个示例 - 可能是新的警报索引页面 -->*/}
- 添加描述好处而不是页面部分/元素的标题
- 描述此功能如何适应/与应用程序的其他部分交互，如果适用（例如，此页面有一个按钮，可将您带到发现查询）。

如果您使用此方法，您可能需要在WHAT/WHY部分中包含一个非常高层次的、一两句话的页面外观描述。

## 链接

- 创建一个名为"了解更多"的部分
- 使用 `<PageGrid />` 元素链接到此功能的子页面

  - 这将显示带有其前端页面描述的页面链接的项目符号列表
  - 这可能需要您编辑页面网格中显示的页面的前端描述

## 包含文件

您可能需要使用 `include` 文件来处理可重用内容的小片段，例如早期采用者注释、计划/功能注释或文本的简短片段。这些可以在 [`/includes/`](https://github.com/getsentry/sentry-docs/tree/master/includes) 中找到。

## 其他指南

以下是一些需要记住的快速提示：

- 将应用程序中的功能主页面称为 _页面_、_主页_ 或甚至 _主页面_。不要称其为 _索引页面_ 或使用 _homepage_（一个单词）。
- 将组织称为 _Sentry_；将应用程序的UI称为 [sentry.io](https://sentry.io)（小写"s"和嵌入式链接），而不是 _Sentry UI_。
- 避免以 [sentry.io](https://sentry.io) 开头句子。
- 不要跳过标题级别。不要将H3作为您的第一个子标题/页面级别标题。页面标题被视为H1，之后应使用的下一个标题级别是H2。如果您跳过标题级别，右侧边栏菜单将无法呈现。
- 不要在页面上添加更多H1标题。标题是H1。页面上的所有其他标题应为H2或更低层次（即数字更高）。

指纹规则（以前称为_服务器端指纹_）使用类似于[堆栈跟踪规则](../stack-trace-rules/)的配置进行配置，但语法略有不同。匹配器相同，但代替翻转标志，分配指纹并完全覆盖默认分组。分配指纹也可以细化默认分组，而不是覆盖它，如果指纹包含值 `{{ default }}`。

这些规则可以在**项目设置 > 问题分组 > 指纹规则**中按项目配置。此设置具有输入字段，您可以在其中编写自定义指纹规则。要更新规则：

1. 确定将问题分组在一起的匹配逻辑。
1. 设置匹配逻辑及其指纹。

指纹规则的语法类似于[**Discover** 查询](/concepts/search/#syntax)中的语法。要否定匹配，请在表达式开头添加感叹号 (`!`)。完整的语法见[此处](https://github.com/getsentry/sentry/blob/90f5cdfa9ebaf0bfdea63852812a6efc90f13691/src/sentry/grouping/fingerprinting/__init__.py#L36-L73)。

Sentry 尝试匹配指纹中配置的所有值。在堆栈跟踪的情况下，考虑所有帧。如果事件数据与匹配器和表达式的行中的所有值匹配，则应用指纹。当存在多个匹配事件的规则时，应用列表中的第一个匹配规则。

```bash {tabTitle:指纹配置}
# 您可以使用注释来解释规则。规则本身遵循
# 以下语法：
匹配器:表达式 -> 值列表
# 值列表可以是硬编码值或替换值。
```

以下是一个实际示例，它将特定类型的异常分组在一起：

```bash {tabTitle:指纹配置}
error.type:DatabaseUnavailable -> system-down
error.type:ConnectionError -> system-down

error.value:"connection error: *" -> connection-error, {{ transaction }}
```

现在，所有具有错误类型 `DatabaseUnavailable` 或 `ConnectionError` 的事件将被分组到类型为 `system-down` 的问题中。此外，所有具有错误值 `connection error` 的事件将按其事务名称分组。因此，例如，如果您的事务 `/api/users/foo/` 和 `/api/events/foo/` - 都具有值 `connection error` - 以相同的方式崩溃，Sentry 将创建两个问题，无论堆栈跟踪或任何其他默认分组方法如何。

## 匹配器

匹配器允许您使用[通配符模式](<https://en.wikipedia.org/wiki/Glob_(programming)>)。

如果要创建包含方括号 `[ ]`、大括号 `{ }` 或文字 `*` 的规则，请用反斜杠转义它们；即 `\[`、`\]`、`\{`、`\}` 或 `\*`。

以下匹配器可用：

<DefinitionList>

### `error.type`

别名：`type`

匹配异常类型（异常名称）。匹配作为区分大小写的通配符执行。

```bash {tabTitle:指纹配置}
error.type:ZeroDivisionError -> zero-division
error.type:ConnectionError -> connection-error
```

### `error.value`

别名：`value`

匹配异常值。错误或异常通常具有与之关联的人类可读描述（值）。此匹配器允许不区分大小写的匹配。

```bash {tabTitle:指纹配置}
error.value:"connection error (code: *)" -> connection-error
error.value:"could not connect (*)" -> connection-error
```

### `level`

匹配日志级别。匹配不区分大小写。

```bash {tabTitle:指纹配置}
logger:"com.myapp.FooLogger" level:"error" -> mylogger-error
```

### `logger`

匹配记录器的名称，这对于将记录器的所有消息组合在一起很有用。此匹配区分大小写。

```bash {tabTitle:指纹配置}
logger:"com.myapp.mypackage.*" -> mypackage-logger
```

### `message`

匹配日志消息。它还将自动检查附加的异常值，因为它们可能很难区分。匹配不区分大小写。

```bash {tabTitle:指纹配置}
message:"system encountered a fatal problem: *" -> fatal-log
```

### `stack.abs_path`

别名：`path`

匹配事件的路径，不区分大小写。它使用路径通配符语义，这意味着 `*` 不匹配斜杠，但 `**` 匹配。请注意，此匹配器匹配 `abs_path` 和 `filename`，因为 SDK 对这些值的支持可能非常不一致。如果通配符匹配这些值中的任何一个，则视为匹配。

```bash {tabTitle:指纹配置}
stack.abs_path:"**/my-utils/*.js" -> my-utils, {{ error.type }}
```

### `stack.function`

别名：`function`

检查堆栈跟踪中的任何函数是否与通配符匹配。匹配区分大小写：

```bash {tabTitle:指纹配置}
stack.function:"my_assertion_failed" -> my-assertion-failed
```

### `stack.module`

别名：`module`

类似于 `stack.abs_path` 但匹配模块名称。匹配区分大小写，并且应用常规通配符规则（`*` 也匹配斜杠）。

```bash {tabTitle:指纹配置}
stack.module:"*/my-utils/*" -> my-utils, {{ error.type }}
```

### `stack.package`

别名：`package`

匹配帧的"包"。这通常是包含帧的调试符号/对象文件的名称。如果任何帧匹配此对象文件，则匹配。

```bash {tabTitle:指纹配置}
stack.package:"**/libcurl.dylib" -> libcurl
stack.package:"**/libcurl.so" -> libcurl
```

### `tags.tag_name`

匹配标签 `tag_name` 的值。这对于过滤某些类型的事件很有用。例如，您可以分离由特定服务器引起的事件：

```bash {tabTitle:指纹配置}
tags.server_name:"canary-*.mycompany.internal" -> canary-events
```

此外，以下实用程序匹配器可用：

### `app`

检查帧是否在应用程序内。当与另一个匹配器组合时特别有用。可能的值为 `yes` 和 `no`：

```bash {tabTitle:指纹配置}
app:yes stack.function:"assert" -> assert
```

### `family`

用于"限定"匹配器的范围。以下系列存在：`javascript` 用于任何类型的 JavaScript 事件，`native` 用于任何类型的原生事件。任何其他平台都称为 `other`。

```bash {tabTitle:指纹配置}
family:native !stack.module:"myproject::*" -> not-from-my-project
```

</DefinitionList>

## 组合匹配器

当组合多个匹配器时，它们都需要匹配。对帧进行操作的匹配器都必须应用于同一帧；否则，它们不被视为匹配。

例如，如果您同时匹配函数名称和模块名称，则仅当帧匹配_两者_函数名称和模块名称时才存在匹配。帧仅匹配函数名称是不够的，即使另一个帧本身会匹配模块名称。

```bash {tabTitle:指纹配置}
# 如果存在具有特定函数和模块名称的帧，则匹配
# 并且还抛出特定错误类型
error.type:ConnectionError stack.function:"connect" stack.module:"bot" -> bot-error
```

## 变量

在指纹的右侧，您可以使用常量和变量。变量会自动替换，并且具有与匹配器相同的名称，但它们可能以不同的方式填充。这些变量也可以在分配指纹时使用 Sentry SDK 时使用。

变量用双大括号 (`{{ variable_name }}`) 括起来。

<DefinitionList>

### `{{ default }}`

这将填充正常分组操作产生的默认指纹。如果您想按其他内容细分已存在的组，这很有用：

```bash {tabTitle:指纹配置}
stack.function:"query_database" -> {{ default }}, {{ transaction }}
```

### `{{ error.type }}`

别名：`{{ type }}`

这将填充发生的错误的名称。当使用链式异常时，它将是最近抛出的错误。这将强制按错误类型创建组：

```bash {tabTitle:指纹配置}
stack.function:"evaluate_script" -> script-evaluation, {{ error.type }}
```

### `{{ error.value }}`

别名：`{{ value }}`

这将填充发生的错误的字符串化值。当使用链式异常时，它将是最近抛出的错误。请注意，当错误值频繁更改时，这可能会产生非常糟糕的组。

```bash {tabTitle:指纹配置}
error.type:"ScriptError" -> script-evaluation, {{ error.value }}
```

### `{{ level }}`

这将填充用于创建事件的日志级别的名称。

```bash {tabTitle:指纹配置}
message:"connection error*" -> connection-error, {{ logger }}, {{ level }}
```

### `{{ logger }}`

这将填充导致事件的记录器的名称。

```bash {tabTitle:指纹配置}
message:"critical connection error*" -> connection-error, {{ logger }}
```

### `{{ message }}`

这将填充事件的消息（类似于 `error.value`，但对于捕获的消息）。请注意，如果消息频繁更改，这可能会产生数据质量差的组：

```bash {tabTitle:指纹配置}
logger:"com.foo.auditlogger.*" -> audit-log, {{ message }}
```

### `{{ stack.abs_path }}`

别名：`{{ path }}`

这将填充"崩溃帧"的路径，也称为应用程序代码的顶层帧。

```bash {tabTitle:指纹配置}
error.type:"ScriptError" -> script-evaluation, {{ stack.abs_path }}
```

### `{{ stack.filename }}`

这类似于 `stack.abs_path`，但只会填充相对文件名：

```bash {tabTitle:指纹配置}
error.type:"ScriptError" -> script-evaluation, {{ stack.filename }}
```

### `{{ stack.function }}`

别名：`{{ function }}`

这将填充"崩溃帧"的函数名称，也称为应用程序代码的顶层帧。

```bash {tabTitle:指纹配置}
error.type:"ScriptError" -> script-evaluation, {{ stack.function }}
```

### `{{ stack.module }}`

别名：`{{ module }}`

这将填充"崩溃帧"的模块名称，也称为应用程序代码的顶层帧。

```bash {tabTitle:指纹配置}
error.type:"ScriptError" -> script-evaluation, {{ stack.module }}
```

### `{{ stack.package }}`

别名：`{{ package }}`

这将填充"崩溃帧"的包名称（对象文件），也称为应用程序代码的顶层帧。

```bash {tabTitle:指纹配置}
stack.function:"assert" -> assertion, {{ stack.package }}
```

### `{{ tags.tag_name }}`

这将标签的值填充到指纹中，例如，可用于按服务器名称或其他类似内容拆分事件。

```bash {tabTitle:指纹配置}
message:"connection error*" -> connection-error, {{ tags.server_name }}
```

### `{{ transaction }}`

这将事务的名称填充到指纹中。它将强制按事务创建组：

```bash {tabTitle:指纹配置}
error.type:"ApiError" -> api-error, {{ transaction }}
```

</DefinitionList>

## 自定义标题

当您使用指纹将事件分组在一起时，有时更改事件的默认标题也很有用。通常，事件的标题是异常类型和值（或某些平台的顶层函数名称）。当您按自定义规则分组时，此标题通常会产生误导。例如，如果您将记录器分组在一起，您可能希望以该记录器命名组。这可以通过设置 `title` 属性来完成，如下所示：

```bash {tabTitle:指纹配置}
logger:my.package.* level:error -> error-logger, {{ logger }} title="来自记录器 {{ logger }} 的错误"
```

<Include name="only-error-issues-note.mdx" />

堆栈跟踪规则通过确保准确分组和更好地将堆栈帧分类为应用程序内或系统，从而改善问题跟踪。这有助于专注于相关代码，减少噪音，并最小化误报。通过为您的项目定制规则，您可以简化调试并在团队或多个应用程序之间保持一致性。

当您在 Sentry 中为分组设置堆栈跟踪规则（以前称为_分组增强_）时，它们会影响输入到分组算法中的数据。这些规则可以在**项目设置 > 问题分组 > 堆栈跟踪规则**中按项目配置。

以下是关于堆栈跟踪规则需要注意的几点：

- 每个规则写在单独一行上。
- 规则由一个或多个匹配表达式组成，后跟当所有表达式匹配时触发的一个或多个操作。
- 规则按顺序从上到下应用于堆栈跟踪中的所有帧。

此外，使用以下匹配器和操作的堆栈跟踪规则也可以应用于传入配置文件以改善帧分类（例如应用程序内与系统）。

允许的匹配器：
  * `stack.abs_path`
  * `stack.module`
  * `stack.function`
  * `stack.package`

允许的操作：
  * `+app`
  * `-app`

堆栈跟踪规则的语法类似于：

```bash {tabTitle: 增强规则}
匹配器名称:表达式 其他匹配器:表达式... 操作1 操作2...
```

如果要否定匹配，请在表达式前加上感叹号 (`!`)。如果行以哈希 (`#`) 开头，则忽略它并将其视为注释。

以下是实际示例：

```bash {tabTitle: 增强规则}
# 将 std 命名空间中的所有函数标记为在应用程序外
family:native stack.function:std::*       -app

# 将 node_modules 中的所有代码标记为不在应用程序内
stack.abs_path:**/node_modules/**         -app

# 从所有分组中删除所有生成的 JavaScript 代码
stack.abs_path:**/generated/**.js         -group
```

## 匹配器

可以在一行中定义多个匹配器。以下匹配器可用：

<DefinitionList>

### `family`

匹配一般平台系列，目前包括 `javascript`、`native` 和 `other`。用逗号分隔规则以将它们应用于多个平台。

```bash {tabTitle: 匹配多个}
family:javascript,native stack.abs_path:**/generated/**  -group
```

```bash {tabTitle: 仅匹配 JavaScript}
family:javascript stack.abs_path:**/generated/**  -group
```

```bash {tabTitle: 仅匹配 native}
family:native stack.abs_path:**/generated/**  -group
```

### `stack.abs_path`

别名：`path`

此匹配器在堆栈跟踪中的路径上不区分大小写，具有 Unix 通配符行为。路径分隔符规范化为 `/`。作为特殊规则，如果文件名是相对的，它仍然匹配 `**/`。

```bash {tabTitle: 增强规则}
# 匹配 `project` 下所有具有 .c 扩展名的文件
stack.abs_path:**/project/**.c +app

# 匹配 vendor/foo 而不带子文件夹
stack.abs_path:**/vendor/foo/*.c -app

# 匹配 `foo.gen.c` 以及 `foo/bar.gen.c`。
stack.abs_path:**/*.gen.c -group
```

### `stack.module`

别名：`module`

模块类似于 `path`，但匹配 `module`。这不用于 Native，但用于 JavaScript、Python 和类似平台。匹配区分大小写，并且可以使用普通通配符。请注意，模块不是包，这对于 Native 环境可能会造成混淆。

### `stack.function`

别名：`function`

匹配堆栈跟踪中的函数，区分大小写，具有普通通配符。

```bash {tabTitle: 增强规则}
stack.function:myproject_* +app
stack.function:malloc      -group
```

### `stack.package`

别名：`package`

匹配堆栈跟踪中的包。包是包含函数或模块的容器。这是 `.jar`、`.dylib` 或类似文件。应用与 `path` 相同的匹配规则。例如，这通常是绝对路径。

```bash {tabTitle: 增强规则}
stack.package:**/libcurl.dylib -group
```

### `app`

匹配堆栈跟踪帧的应用程序内标志的当前状态。`yes` 表示帧在应用程序内，`no` 表示不在。

### `category`

匹配[内置](#built-in-categories)或用户定义的帧类别。
有关如何设置类别，请参阅[变量操作](#variable-actions)。

```bash {tabTitle: 类别匹配}
category:telemetry -group
```

#### 内置类别

帧分类被我们的最新分组算法广泛使用。您可以匹配各种类别，包括：

- `system` - 检测到的系统库
- `std` - 检测到的标准库
- `ui` - UI 框架
- `driver` - 图形驱动程序等
- `telemetry` - 崩溃报告和分析框架

在我们的[源代码](https://github.com/getsentry/sentry/blob/master/src/sentry/grouping/enhancer/enhancement-configs/mobile%402021-04-02.txt)中查看内置类别的完整列表。

</DefinitionList>

### 匹配兄弟帧

如果需要有关周围帧的信息才能将规则应用于帧，请使用调用者/被调用者匹配语法。例如：

```bash {tabTitle: 兄弟匹配}
# 如果它们由遥测调用，则忽略应用程序内帧
[ category:telemetry ] | app:yes -group

# 如果它们调用应用程序帧，则忽略系统帧
category:system | [ app:yes ] -group
```

## 操作

有两种类型的**操作**：标志和变量设置。

### 标志操作

_标志_标识当所有匹配器匹配时要执行的操作，它使用这些前缀：

- `+` 设置标志
- `-` 取消设置标志
- `^` 应用于匹配帧上方的帧（朝向崩溃）
- `v` 应用于匹配帧下方的帧（远离崩溃）

例如，`-group ^-group` 从分组中删除匹配帧及其上方的所有帧。

- `app`：标记或取消标记帧为应用程序内
- `group`：从分组中添加或删除帧

### 变量操作

可以定义有限的_变量_集 (`variable=value`)：

- `max-frames`：设置用于分组的总帧数。默认为 `0`，表示"所有帧"。如果设置为 `3`，则仅考虑顶部三帧。

{/*<!--
- `category`：为帧设置自定义类别。

  - 自定义类别覆盖[内置类别](#built-in-categories)。
  - 如果多个应用类别的规则匹配帧，则仅保留最底部规则中设置的类别。
-->*/}

### 示例

```bash {tabTitle: 增强规则}
stack.abs_path:**/node_modules/** -group
stack.abs_path:**/app/utils/requestError.jsx -group
stack.abs_path:**src/getsentry/src/getsentry/** +app

family:native max-frames=3

stack.function:fetchSavedSearches v-group
stack.abs_path:**/app/views/**.jsx stack.function:fetchData ^-group

family:native stack.function:SpawnThread v-app -app
family:native stack.function:_NSRaiseError ^-group -app
family:native stack.function:std::* -app
family:native stack.function:core::* -app
```

## 建议

这些建议将大大改善您的开箱即用分组体验。

### 标记应用程序内帧

为了主动改善您的体验，帮助 Sentry 确定堆栈跟踪中的哪些帧是"应用程序内"（您自己的应用程序的一部分），哪些不是。SDK 定义了默认规则，但在许多情况下，这也可以在服务器上进行改进。特别是对于需要服务器端处理的语言（例如，Native C、C++ 或 JavaScript），最好在服务器上覆盖此设置。

例如，以下将所有低于特定 C++ 命名空间的帧标记为应用程序内：

```bash {tabTitle: 增强规则}
stack.function:myapplication::* +app
```

查看[Apple 的应用程序内帧](/platforms/apple/usage/in-app-frames)以了解 `sentry-cocoa` SDK 如何将帧标记为应用程序内。

#### 堆栈跟踪规则

以下将以 `_dispatch_` 开头的 libdispatch 帧标记为 `inApp`。

```bash {tabTitle: 增强规则}
stack.function:_dispatch_*  +app
```

您还可以通过将其他帧标记为"不在应用程序内"来实现相同的结果。但是，如果是这种情况，您应该首先确保所有帧都设置为"应用程序内"以覆盖默认值：

```bash {tabTitle: 增强规则}
app:no                   +app
stack.function:std::*    -app
stack.function:boost::*  -app
```

<Alert>

您需要首先强制所有帧为应用程序内，因为客户端 SDK 或早期处理可能已经设置了一些默认值。

</Alert>

如果您使⽤ `sentry-cocoa` SDK，您还可以通过将其他帧标记为 `not inApp` 来实现相同的结果。以下示例将 `DataRequest` 和 `DownloadRequest` 类中的函数标记为 `not inApp`。

```bash {tabTitle: 增强规则}
stack.function:DataRequest*  -app
stack.function:DownloadRequest*  -app
```

### 剪切堆栈跟踪

在许多情况下，您希望删除堆栈跟踪的顶部或底部。例如，许多代码库使用公共函数来⽣成错误。在这种情况下，错误机制将显示为堆栈跟踪的⼀部分。

例如，如果您使⽤ Rust，您可能希望删除与恐慌处理相关的⼀些帧：

```bash {tabTitle: 增强规则}
stack.function:std::panicking::begin_panic       ^-app -app ^-group -group
stack.function:core::panicking::begin_panic      ^-app -app ^-group -group
```

在这⾥，我们告诉系统从 `begin-panic` 到崩溃位置的所有帧都不是应⽤程序的⼀部分（包括恐慌帧本身）。在所有情况下，上⽅的帧都与分组⽆关。

同样，您还可以删除堆栈跟踪的基础。如果您有不同的主循环驱动应⽤程序，这特别有用：

```bash {tabTitle: 增强规则}
stack.function:myapp::LinuxMainLoop         v-group -group
stack.function:myapp::MacMainLoop           v-group -group
stack.function:myapp::WinMainLoop           v-group -group
```

### 堆栈跟踪帧限制

这对_所有_项⽬都没有⽤，但对于有许多崩溃的⼤型应⽤程序可能很有效。默认策略是考虑⼤部分堆栈跟踪与分组相关。这意味着导致崩溃函数的每个不同堆栈跟踪都会导致创建不同的组。如果您不希望这样，您可以通过限制应考虑多少帧来强制组变得⼤得多。

例如，如果堆栈跟踪中的任何帧引⽤公共外部库，您可以告诉系统仅考虑顶部 N 帧：

```bash {tabTitle: 增强规则}
# 始终仅考虑所有原生事件的顶部 1 帧
family:native max-frames=1

# 如果错误在 proprietarymodule.so 中，仅考虑顶部 2 帧
family:native stack.package:**/proprietarymodule.so  max-frames=2

# 对于这些函数，我们希望考虑更多堆栈跟踪
family:native stack.function:KnownBadFunction1  max-frames=5
family:native stack.function:KnownBadFunction2  max-frames=5
```

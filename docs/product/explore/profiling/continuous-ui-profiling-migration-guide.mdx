要了解之前基于事务的和新的连续性能分析和 UI 性能分析产品之间的概念差异，请参见[事务与连续性能分析文档](/product/explore/profiling/transaction-vs-continuous-profiling/)。简而言之，主要区别是：

- 基于事务的性能分析绑定到追踪的生命周期。性能分析仅在事务或根跨度期间发生，不能独立启动。**连续和 UI 性能分析支持追踪绑定的生命周期模式，以及允许您使用开始/停止 API 直接控制性能分析器的手动生命周期模式**。
- 基于事务的性能分析限制为 30 秒长度。**连续和 UI 性能分析支持任意长度的性能分析。**

本指南将重点介绍如何将您的插桩从基于事务的性能分析迁移到连续性能分析和 UI 性能分析，并解释迁移的计费影响。我们计划最终弃用基于事务的 API，因此我们建议您尽快迁移到新的 API。

对于本文档的其余部分，您可以假设在引用"连续/UI 性能分析"的任何地方，相同的指导适用于连续性能分析（面向后端）和 UI 性能分析（面向前端）产品，即使它们针对不同的用例。

## 可用性

连续性能分析在以下 SDK 上受支持：

- [Python](/platforms/python/profiling/) `2.24.1`
- [Node.js](/platforms/javascript/guides/node/profiling/) `9.8.0`

UI 性能分析在以下 SDK 上受支持：

- [iOS 和 macOS](/platforms/apple/profiling/) `8.49.0`
- [Android（Java 和 Kotlin 仅限）](/platforms/android/profiling/) `8.5.0`

如果您正在使用任何其他当前支持基于事务的性能分析的 SDK，这些 SDK 尚不支持连续或 UI 性能分析。我们将随着时间的推移为更多 SDK 添加支持。

## 识别当前 API 使用情况

如果您正在指定 `profiles_sample_rate` 或 `profiles_sampler` 配置选项，您正在使用基于事务的性能分析。

如果您正在指定 `profile_session_sample_rate` 和/或 `profile_lifecycle` 配置选项，您正在使用连续性能分析或 UI 性能分析。

## 迁移到连续/UI 性能分析

### 1. 用 `profile_session_sample_rate` 替换 `profiles_sample_rate` 和 `profiles_sampler` SDK 配置选项

在基于事务的和连续/UI 性能分析之间，客户端采样的实现方式存在重要区别。`profiles_sample_rate` 和 `profiles_sampler` 用于评估是否对每个*事务*采样性能分析。换句话说，如果 `profiles_sample_rate` 设置为 0.5，50% 的所有事务将有关联的性能分析。`profiles_sampler` 的工作方式类似——唯一的区别是您可以根据被采样事务的上下文实现自定义采样逻辑（而不是固定采样率）。

相比之下，连续/UI 性能分析使用*会话范围*的客户端采样，通过新的 `profile_session_sample_rate` 参数配置。采样在每个会话开始时评估一次，而不是每次收集性能分析数据时评估。由于应用采样的范围不同，您不能直接将之前的 `profiles_sample_rate` 值转换为 `profile_session_sample_rate`。

要控制客户端采样行为，请将 `profile_session_sample_rate` 配置参数设置为 0.0-1.0 的值（默认为 0.0）以控制采样的性能分析会话数量。此采样率的应用方式取决于上下文：

- **对于连续性能分析**：会话在 Sentry SDK 初始化时开始，在服务或进程终止时停止。采样仅在进程生命周期内评估一次（在初始化期间）。
    - 例如，如果您将连续性能分析与后端服务一起使用，其中部署了 N 个相同服务的实例（如 N 个容器、pod 等），`profile_session_sample_rate` 控制这些实例中将启用性能分析的百分比。0.5 将意味着随机采样的 50% 的 N 个实例将启用性能分析。
    - 采样仅在实例重新启动或重新部署时重新评估
- **对于 UI 性能分析**：初始用户会话在 Sentry SDK 初始化时开始，此时首次评估采样。用户会话将在应用程序终止时结束*或*，取决于平台，可能有其他事件结束用户会话（例如将移动应用程序置于后台）。采样率将在每个新用户会话上重新评估。
    - 例如，在浏览器上，用户会话在标签页打开时开始，在标签页关闭时结束。采样将在每个会话开始时评估一次，因此如果 `profile_session_sample_rate` 设置为 0.5，那么 50% 的时间您的应用程序在标签页中打开时，性能分析将在该标签页的生命周期内处于活动状态。
    - 在移动设备上，用户会话在应用程序启动或置于前台时开始，在应用程序终止或置于后台时结束。采样将在每个会话开始时评估一次，因此如果 `profile_session_sample_rate` 设置为 0.5，那么 50% 的时间用户打开您的移动应用程序时，性能分析将处于活动状态，直到用户将应用程序置于后台或退出应用程序。
    
### 2. 设置新的 `profile_lifecycle` SDK 配置选项

连续/UI 性能分析支持两种生命周期模式：`manual` 和 `trace`。

**`manual` 是默认模式**，允许您通过调用 `start_profile_session` 和 `stop_profile_session` 函数更直接地控制性能分析器何时运行。如果您决定使用此模式，您必须在希望性能分析处于活动状态的代码中的任何位置添加这些开始/停止调用。如果未调用 `start_profile_session`，性能分析将永远不会发生。请注意，这遵循 `profile_session_sample_rate`——如果会话未被采样，即使调用 `start_profile_session`，也不会收集性能分析数据。

**如果您想要更自动的性能分析行为，类似于之前的基于事务的性能分析行为，您可以使用 `trace` 生命周期模式。** 在此模式下，您不需要手动调用 `start_profile_session` 和 `stop_profile_session`，因为性能分析将在有活动事务或根跨度时自动开始和停止。此模式遵循 `profile_session_sample_rate`，就像 `manual` 生命周期模式一样，但也遵循 `traces_sample_rate` 和 `traces_sampler`——为了发生性能分析，性能分析会话必须被采样，并且追踪也必须被采样。

## 更新您的订阅

对连续/UI 性能分析的访问仅通过您的即用即付（PAYG）预算提供。要设置 PAYG 预算或对现有 PAYG 预算进行更改，请转到您的[**设置**](https://sentry.io/orgredirect/organizations/:orgslug/settings/)页面并点击"订阅"（在"使用情况和计费"标题下）。有关更多信息，请参见我们的[定价文档](/pricing/)。

您可以通过访问统计页面查看基于事务的性能分析的连续和/或 UI 性能分析小时数，以帮助估计迁移时所需的性能分析小时数。

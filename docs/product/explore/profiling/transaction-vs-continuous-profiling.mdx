我们发布了两个新产品：连续性能分析（专注于后端应用程序）和 UI 性能分析（专注于前端应用程序），旨在取代我们之前的基于事务的性能分析产品。继续阅读以了解这些实现之间的差异。

## 基于事务的性能分析的历史背景

我们对性能分析的初始实现是我们整篇文档中称为*基于事务的性能分析*的内容。顾名思义，在这种模式下，性能分析与事务的生命周期耦合（我们用来描述[根跨度](/concepts/key-terms/tracing/)的术语）。当事务开始时，性能分析自动开始，当事务完成时，性能分析自动完成。这意味着如果您的应用程序已经用追踪进行了插桩，性能分析可以用最少的额外工作启用。虽然这种方法简化了设置，但它带来了一些缺点。

一个主要限制是性能分析不能超过 30 秒的持续时间。由于性能分析必须与其对应的事务一起发送，SDK 需要在客户端上为事务的整个持续时间缓冲性能分析数据。事务可以是任意长的，但性能分析有效负载与跨度相比很大，因此我们不能无限期地缓冲性能分析数据，而不冒客户端上过度内存使用和 OOM（内存不足）状态的风险。由于这个限制，不可能将基于事务的性能分析用于长时间运行的操作（例如，在前端分析完整的用户会话，或在后端分析长时间运行的任务）

另一个限制是通过与事务耦合，性能分析仅限于应用程序中具有足够追踪插桩覆盖的部分。如果应用程序的某些部分没有用跨度进行插桩，也不会收集性能分析数据。这些数据缺口的存在使得很难准确确定最严重的性能瓶颈是什么。

连续性能分析和 UI 性能分析旨在解决这些限制，并提供在更广泛场景中更有用的更强大的性能分析功能。**这些产品对最大性能分析长度没有上限，因为性能分析数据是分块流式传输的，它们提供对性能分析器何时运行的完全控制。**

## 连续性能分析

连续性能分析产品旨在帮助调试后端应用程序和服务中的性能瓶颈。使用连续性能分析的推荐方法是在服务启动时启动性能分析器，在服务终止时停止性能分析。通过在服务的整个生命周期内连续运行性能分析器，Sentry 可以提供对可能导致过度资源使用从而导致高基础设施成本的代码路径的完全可见性。

如果您担心成本或[性能开销](/product/explore/profiling/performance-overhead)，可以通过多种方式限制连续性能分析的使用：

* 使用客户端采样控制来收集相同服务的较少实例的性能分析（例如，仅分析服务的 10% 实例）
* 使用 `start_profiler` 和 `stop_profiler` API 仅选择性地分析已知性能关键的服务部分

## UI 性能分析

UI 性能分析产品旨在帮助调试前端应用程序中的性能瓶颈。这些瓶颈通常会导致用户面对的问题，如卡顿的滚动或动画、高交互延迟和缓慢的加载时间。使用 UI 性能分析的推荐方法是在用户会话开始时启动性能分析器，在用户会话结束时停止性能分析（当应用程序终止时，或如果在移动设备上，当应用程序置于后台时）。通过分析完整的用户会话，Sentry 可以提供对可能降低用户体验并在用户设备上导致过度电池使用的代码路径的完全可见性。

如果您担心成本或[性能开销](/product/explore/profiling/performance-overhead)，可以通过多种方式限制 UI 性能分析的使用：

* 使用客户端采样控制来收集较少用户会话的性能分析（例如，仅分析所有用户会话的 10%）
* 使用 `start_profiler` 和 `stop_profiler` API 仅选择性地分析已知性能关键的应用程序部分

## 生命周期模式：手动和追踪

连续性能分析和 UI 性能分析公开了两种新的生命周期模式，帮助您精确控制性能分析在应用程序中的行为。

**手动**生命周期模式通过使用新的 `start_profiler` 和 `stop_profiler` API 提供对性能分析器何时运行的完全控制。这是默认的生命周期模式。如果未调用 `start_profiler`，将永远不会收集性能分析数据。

基于事务的性能分析的好处之一是能够自动收集性能分析与追踪，而无需手动插桩代码库。**追踪**生命周期模式提供类似于旧的基于事务的性能分析行为的行为，其中性能分析将在有活动事务或根跨度时自动开始和停止。此生命周期模式不需要手动调用 `start_profiler`，但确实需要启用追踪，并且应用程序已用跨度进行插桩。

## 采样

与基于事务的性能分析一样，连续性能分析和 UI 性能分析支持客户端采样控制。但是，采样现在发生在**会话**范围而不是单个性能分析范围。有关采样如何工作的更多信息，请阅读[迁移指南](/product/explore/profiling/continuous-ui-profiling-migration-guide/)。

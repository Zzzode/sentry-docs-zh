有多种方法可以可视化和使用性能分析数据，包括差异火焰图，它可以帮助轻松识别函数回归或执行上下文的变化。

## 什么是差异火焰图？

差异火焰图是图形可视化，突出显示程序调用堆栈频率的变化，这是一项没有它们时可能很乏味的任务。它们通常用于分析中查找函数回归并了解哪些函数变慢、变快或新添加。请记住，差异火焰图并不是一个真正的新概念，而只是使用由两个差异生成的颜色绘制的聚合火焰图。

在 Sentry 中，当检测到函数回归时，会显示差异火焰图。当发生回归时，Sentry 会生成两个聚合火焰图。一个来自回归前的时期，另一个来自回归后的时间。然后使用差异火焰图来可视化两者之间的差异。

<Alert>

在差异火焰图中，红色意味着函数的执行时间增加或存在新出现的函数。蓝色意味着执行时间减少。

</Alert>

在解释差异火焰图时，请留意红色框架。它们是您的应用程序代码性能变差的原因。

![差异火焰图锁竞争](./img/differential-flamegraph.png)

上面的差异火焰图显示了对 `ModuleLock.acquire` 的函数调用持续时间增加，暗示了减慢代码速度的锁竞争问题。

让我们看另一个例子：

![差异火焰图新函数](./img/differential-flamegraph-new.png)

上面的差异火焰图显示了对 `OrganizationTeamsEndpoint.dispatch` 的新函数调用被引入，这使代码运行变慢，而对 `ChunkUploadEndpoint.get` 的调用减少。

如果您仔细观察，您会注意到右上角有一个 `Before -> After` 和 `After -> Before` 切换。这是因为默认情况下，差异火焰图将回归发生后发生的聚合火焰图作为事实来源。这意味着任何可能已被删除的代码将不再被绘制（毕竟，它已经不存在了）。这就是为什么否定视图很有用，因为它允许我们比较之前的数据，并窥视我们的代码将如何变化的未来。

```
以下是我们使用回归发生后发生的性能分析数据作为事实来源时，模块锁问题的示例：

![差异火焰图新函数](./img/differential-flamegraph-regular.png)

如果我们使用回归发生前的数据作为事实来源，它看起来是这样的：

![差异火焰图新函数](./img/differential-flamegraph-inverted.png)

通过使用 `Before -> After` 切换，我们可以看到 `ModuleLock.acquire` 在回归前更快，但我们也可以看到我们 `ChunkUploadEndpoint` 内部的一些函数调用，这些调用不再被调用。
```

虽然 Sentry 中的差异火焰图使用与常规火焰图不同的配色方案并从不同的数据源提取，但您可以像使用常规火焰图一样在图表中平移和缩放并搜索框架。

差异火焰图下方的表格显示函数持续时间中最大增加和最大减少。

![差异火焰图表格](./img/differential-flamegraph-table.png)

差异火焰图是快速可视化性能回归或改进的潜在来源的好方法。目前，它们仅作为自动函数回归工作流的一部分提供，但我们正在努力将它们添加到性能分析工作流的其他部分。

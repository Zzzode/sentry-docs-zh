会话重放通过观察和记录您的 Web 应用程序的 DOM 变化并通过公共互联网将该数据传输到事件接收服务器来工作。为了在不负面影响主机页面性能的情况下执行此工作，会话重放 SDK 注意引入最小的额外文件大小，以非侵入式方式观察和记录 DOM 内容，发送所需的最小字节数，并使用地理上接近最终用户的低延迟接收端点。

**对于大多数 Web 应用程序，我们的客户端 SDK 的性能开销对最终用户来说是不可察觉的。**

## 会话重放如何优化？

Sentry 的会话重放 SDK 采取了几项措施来避免对运行它的页面性能产生负面影响：

- 从版本 [7.78.0](https://blog.sentry.io/sentry-bundle-size-how-we-reduced-replay-sdk-by-35/) 开始，会话重放插件是额外的约 36KB gzip 压缩。
  - 在我们自己的研究中，这是可用的最小文件大小的会话重放 SDK 之一。
  - 我们提供了多种加载重放包的方式，包括[通过我们的加载器脚本](/platforms/javascript/install/loader/)。
- 会话重放 SDK 通过快照网页的文档对象模型（DOM）并将其内容传输到 Sentry 的服务器来工作。为了最小化传输的字节数，采用了多种策略：
  - 一旦初始 DOM 树被快照，SDK 记录并传输 DOM 的变化（增量）而不是再次快照整个树。这确保了传输到重放的最小状态量。此外，SDK 本身不花费 CPU 周期扫描变化；相反，它监听由 MutationObserver 发出的变化，这是一个内部浏览器 API。
  - DOM 内容本身在通过 HTTP 传输之前在客户端进行 gzip 压缩。压缩操作使用 Web Worker 在后台线程中执行，这意味着浏览器的 UI 线程不受影响。
  - SDK 不会导致浏览器下载和传输其他静态资产，如图像、视频等。相反，在重放期间，这些资产直接从原始主机服务器获取。
- Sentry 的事件接收基础设施使用分布式存在点（PoPs），将接收服务器放置在世界各地并靠近您的用户。当会话重放事件被传输时，用户的浏览器连接到并传输事件有效负载到其区域可用的最近 PoP。这大大减少了端到端延迟，并最小化了放置在浏览器上的网络开销。
- SDK 被设计为在需要时优雅地降级。在一次性发生数千个 DOM 突变的情况下，SDK 将禁用录制以确保录制网站不会进一步降低用户体验。突变限制值是可配置的。

## 性能开销的因素

虽然我们的客户端 SDK 的性能开销在大多数时候对最终用户来说是不可察觉的，但它可能基于应用程序的复杂性而变化。如果应用程序具有大型 DOM 和大量 DOM 节点突变，与更简单、大部分静态的网站相比，将有更高的开销。

网页有不同的生命周期阶段，每个阶段都有不同的性能特征。以下是生命周期阶段的列表，以及需要注意的最重要的性能指标。

### 页面加载阶段

- SDK 包大小是我们优化的内容。确保您启用了树摇，并已更新到最新的 SDK 版本（至少版本 7.78.0）以获得最小的包大小。
- 初始 HTML 页面的大小会影响 SDK 读取初始快照所需的时间。使用您的浏览器开发工具，过滤网络面板以显示文档类型并查看大小，以了解对您网站的影响。一个典型的单页应用程序，显示加载屏幕并需要等待 JavaScript 渲染所有内容，将从初始快照中获得可忽略的开销。
- 所有重放 SDK 数据在通过网络发送之前都会被压缩。HTML 非常可压缩，我们对真实网站的压缩效果感到惊讶。如果您启用了 PII 掩码和阻止（这是默认设置），网络开销将进一步减少。掩码文本被替换为星号，gzip 压缩得很好，而被阻止的内容根本不包含。

### 捕获鼠标/触摸事件

- 鼠标移动、点击/触摸和悬停事件的记录非常小且快速；开销可以忽略不计。
- 我们去重点击事件（同时仍然检测愤怒点击）并对鼠标移动事件使用平滑处理，以平衡您在调试时看到的细节与捕获的冗余数据量。

### 动态内容和 DOM 突变

- DOM 突变的频率高度特定于应用程序。如果没有突变发生，您将不会看到任何从 SDK 传输到 Sentry 的重放数据。另一方面，如果您有定期更新 HTML 的 JavaScript 计时器，您将看到网络上发送的 HTML 变化的稳定流。
- DOM 突变的大小也高度特定于应用程序。一次添加/删除/更改更多 DOM 节点将增加解析和最小化数据所需的时间。但请记住，这里的开销发生在 DOM 更改显示给用户之后（在 MutationObserver 回调内部）。
  SDK 具有突变限制以减轻负面用户体验：更改 750 个节点将在重放中发出警告面包屑，而发出 10000 个以上节点将导致重放结束。

### 页面卸载阶段

- SDK 对页面卸载没有特殊处理；内存中的任何内容都将被浏览器释放，用户将尽快过渡到他们的下一个网页。

## 测量性能开销

获得准确指标的唯一方法是自行测量性能开销。您可以对您的网站和基础设施应用现实的访问模式，并将其与您自己的顶级业务指标相关联。
要了解如何在不部署到生产环境的情况下测量应用程序上会话重放的性能开销，请阅读我们的博客文章：测量会话重放开销。

我们使用博客中的方法在 Sentry 的 Web UI 上测量了重放 SDK 的开销。以下是结果（显示中位数）：

| 指标                           | 无 Sentry | 仅 Sentry SDK | Sentry + 重放 SDK |
| -------------------------------- | -------------- | --------------- | ------------------- |
| 最大内容绘制 (LCP) | 1599.19 毫秒     | 1546.07 毫秒      | 1529.11 毫秒          |
| 累积布局偏移 (CLS)    | 0.40 毫秒        | 0.40 毫秒         | 0.40 毫秒             |
| 首次输入延迟 (FID)          | 1.26 毫秒        | 1.30 毫秒         | 1.50 毫秒             |
| 总阻塞时间 (TBT)        | 2621.67 毫秒     | 2663.35 毫秒      | 3036.80 毫秒          |
| 平均内存                   | 119.26 MB      | 125.12 MB       | 124.84 MB           |
| 最大内存                       | 320.66 MB      | 359.21 MB       | 339.03 MB           |
| 网络上传                   | 21 B           | 3.84 KB         | 272.51 KB           |
| 网络下载                 | 8.06 MB        | 8.09 MB         | 8.07 MB             |

基准测试在 Apple M1 MacBook Pro 上针对远程预览服务器和远程 API 后端运行了 50 次迭代。场景可以概括为加载 Sentry，导航到 Discover，添加 4 列，等待结果，添加另一列，最后第二次等待结果。

基准测试是一个严格的录制场景（Discover 数据表是我们最复杂的之一，关于 DOM 节点和突变）。还运行了一个由导航到四个不同设置页面组成的更简单场景，产生了约 100 毫秒的总 JS 阻塞时间增加。

存在一个已知性能问题，当录制具有许多 DOM 突变的重放时会发生。它通常发生在渲染大型数据网格时。我们正在修复，但与此同时，我们建议您使用阻止功能避免录制可能降低客户性能的大型突变。

在您的发布过程中，您可以添加一个步骤在Sentry中创建发布并将其与链接仓库中的提交关联。完成此操作后，您将能够在**发布详情**页面中看到与发布关联的提交。

如果您使用我们的[发布自动化](/product/releases/setup/release-automation/)选项之一，此步骤应该自动完成。

但是，如果您没有安装仓库集成，您可以使用[Sentry CLI](#using-the-cli)或[我们的API](#using-the-api)向Sentry发送原始提交元数据。

## 将提交与发布关联

当您关联提交后，我们可以将发布中的提交联系在一起，包括：

- 这些提交触及的文件
- 堆栈跟踪中观察到的文件
- 这些文件的作者
- 这些提交解决的问题

### 使用CLI

在此CLI示例中：

- 环境变量配置CLI（有关替代方案，请参见[使用项目](/cli/configuration/#sentry-cli-working-with-projects)）
- `propose-version`子命令自动确定发布ID
- 为组织`my-org`的项目`project1`和`project2`创建标记为`VERSION`的发布
- 由于`--auto`标志，仓库名称自动确定，并且将前一个发布的头提交与当前头提交之间的提交与发布关联。

<OrgAuthTokenNote />

```bash
# 假设您在git仓库中
export SENTRY_AUTH_TOKEN=___ORG_AUTH_TOKEN___
export SENTRY_ORG=___ORG_SLUG___
VERSION=$(sentry-cli releases propose-version)

# 创建发布
sentry-cli releases new -p project1 -p project2 $VERSION

# 将提交与发布关联
sentry-cli releases set-commits --auto $VERSION
```

如果您没有与Sentry组织关联的基于仓库的集成，则`--auto`标志将自动使用本地仓库的git树，并将前一个发布的头提交与当前头提交之间的提交与发布关联。如果这是第一个发布，Sentry将使用最新的10-20个提交，具体取决于集成。此行为可使用`--initial-depth`标志配置。

或者，您可以使用`--local`标志默认启用此行为：

```bash
sentry-cli releases set-commits --local $VERSION
```

为了更好地控制要关联的提交，或者如果您无法在仓库内执行命令，您可以手动指定仓库和范围。

以下示例将`from`和`to`之间的提交（或引用）与当前发布关联，其中`from`是前一个发布的提交。`from`提交是可选的；如果省略，我们将使用前一个发布的提交作为基线：

```bash
sentry-cli releases set-commits --commit "my-repo@from..to" $VERSION
```

<Alert>
  仓库名称`my-repo`应该与您链接仓库时输入的名称匹配，并且应采用`owner-name/repo-name`的形式。
</Alert>

### 使用API

要使用API向Sentry发送提交元数据，请使用[创建发布端点](/api/releases/create-a-new-release-for-an-organization/)。

对您的提交使用以下格式：

```json
{
  "commits": [
    {
      "patch_set": [
        { "path": "path/to/added-file.html", "type": "A" },
        { "path": "path/to/modified-file.html", "type": "M" },
        { "path": "path/to/deleted-file.html", "type": "D" }
      ],
      "repository": "owner-name/repo-name",
      "author_name": "作者姓名",
      "author_email": "author_email@example.com",
      "timestamp": "2018-09-20T11:50:22+03:00",
      "message": "这是提交消息。",
      "id": "8371445ab8a9facd271df17038ff295a48accae7"
    }
  ]
}
```

`patch_set`

提交中已更改文件的列表。指定`patch_set`对于支持可疑提交和建议受让人是必要的。它由两部分组成：

    `path`
    : 文件的路径。支持正斜杠和反斜杠（`'/' '\'`）。

    `type`
    : 在该提交中发生的更改类型。选项包括：
        - `添加 (A)`
        - `修改 (M)`
        - `删除 (D)`

`repository`

提交所属的仓库的完整名称。如果未提供此字段，Sentry将生成以下形式的名称：`u'organization-<organization_id>'`（即，如果组织ID是`123`，则生成的仓库名称将是`u'organization-123'`）。

`author_email`

提交作者的电子邮件是启用建议受让人功能所必需的。

`author_name`

也可以包括提交作者的姓名。

`timestamp`

提交时间戳用于对给定提交进行排序。如果未包含时间戳，提交将保持给定顺序。

`message`

提交消息。

`id`

提交ID。

#### 使用补丁数据创建发布

以下是使用补丁数据的请求示例：

<Alert>

确认您使用的是[身份验证令牌](/api/auth/#auth-tokens)，**而不是**[API密钥](/api/auth/#api-keys)，后者已弃用。

</Alert>

```bash
curl https://sentry.io/api/0/organizations/your-organization-name/releases/ \
  -X POST \
  -H 'Authorization: Bearer <Token>' \
  -H 'Content-Type: application/json' \
  -d '
 {
 "version": "2.0rc2",
 "projects":["project-1","project-2"],
 "commits":[
     {
        "patch_set": [
            {"path": "path/to/added-file.html", "type": "A"},
            {"path": "path/to/modified-file.html", "type": "M"},
            {"path": "path/to/deleted-file.html", "type": "D"}
        ],
        "repository": "owner-name/repo-name",
        "author_name": "作者姓名",
        "author_email": "author_email@example.com",
        "timestamp": "2018-09-20T11:50:22+03:00",
        "message": "这是提交消息。",
        "id": "8371445ab8a9facd271df17038ff295a48accae7"
    }
 ]
 }
 '
```

如果您想更好地控制提交出现的顺序，您可以向我们发送所有提交的列表。这可能如下所示：

```python
import subprocess
import requests

SENTRY_API_TOKEN = <my_api_token>
sha_of_previous_release = <previous_sha>

log = subprocess.Popen([
    'git',
    '--no-pager',
    'log',
    '--no-merges',
    '--no-color',
    '--pretty=%H',
    '%s..HEAD' % (sha_of_previous_release,),
], stdout=subprocess.PIPE)

commits = log.stdout.read().strip().split('\n')

data = {
    'commits': [{'id': c, 'repository': 'my-repo-name'} for c in commits],
    'version': commits[0],
    'projects': ['my-project', 'my-other-project'],
}

res = requests.post(
    'https://sentry.io/api/0/organizations/my-org/releases/',
    json=data,
    headers={'Authorization': 'Bearer {}'.format(SENTRY_API_TOKEN)},
)
```

有关更多信息，请参见[API参考](/api/releases/create-a-new-release-for-an-organization/)。

## 通过提交解决问题

此外，当您开始将提交与发布关联时，您将能够通过在提交消息中包含问题ID来解决问题。您可以在**问题详情**页面顶部的受让人下拉菜单旁边找到问题ID。

提交消息可能如下所示：

```bash
防止用户上的空查询

修复 SENTRY-317
```

当Sentry看到此提交时，我们将在问题中引用该提交，当您在Sentry中创建发布时，我们将将该问题标记为在该发布中已解决。您还可以通过在标题或描述中包含`fixes <SENTRY-SHORT-ID>`来使用拉取请求解决问题。

<Alert>

如果您使用GitHub，您可能启用了阻止Sentry识别用户实际电子邮件地址的隐私设置。要使用建议的所有者功能，请在GitHub的[账户设置](https://github.com/settings/emails)中取消选中"保持我的电子邮件地址私密"。

</Alert>

在许多工具中，性能监控只是关于跟踪网页上的几个关键指标。Sentry 采用了不同的方法。通过设置[追踪](/concepts/key-terms/tracing/)，Sentry 捕获整个应用程序堆栈中每个事务的详细性能数据，并自动将其呈现在各种易于使用但功能强大的功能中，以便您可以快速识别和解决性能问题——所有这些都集中在一个地方。

<Alert>

设置[分布式追踪](/product/sentry-basics/tracing/)以启用性能监控。

</Alert>

## 在 Sentry 中监控性能的方法

Sentry 具有多种用于监控性能的工具，这些工具对您的监控和调试需求的不同方面很有用：

### 1：洞察：对堆栈每一层性能数据的有主见的视图

启用分布式追踪后，您将自动看到一组[洞察](/product/insights/)页面，这些页面为您堆栈的每一层提供性能数据的高级视图。这对于快速查看需要优化的任何主要性能异常值或跟踪跨时间、环境、发布等的潜在回归非常有用。

- [**前端洞察**](/product/insights/frontend)为您提供了一种简单但强大的方式来深入研究常见的前端性能问题，如核心 Web 指标，以及事务性能，如持续时间和错误率。
- [**后端洞察**](/product/insights/backend/)显示您的后端事务性能，包括数据库查询、外部 API 调用、缓存、队列、定时任务等。
- [**移动洞察**](/product/insights/mobile/)跟踪您的移动应用程序的关键性能指标，包括冷启动和热启动时间、TTID 和 TTFD，以及整个应用程序中关键屏幕的加载或渲染时间。
- [**AI 洞察**](/product/insights/ai/)为您提供 LLM 性能的视图。这目前处于测试阶段，仅适用于某些集成。

### 2：性能问题：自动识别全栈瓶颈

除了典型的[错误问题](/product/issues/issue-details/error-issues/)之外，Sentry 还将自动检测常见的性能问题并将其分组为[性能问题](/product/issues/issue-details/performance-issues/)。例如：
- 指示对后端进行不必要往返的 N+1 查询和 API 调用问题
- 可以批处理以获得更好性能的连续数据库查询或 HTTP 请求
- 在主线程上运行的数据库查询或文件 I/O，这可能会阻塞 UI 线程并导致性能问题
- 以及更多记录在[此处](/product/issues/performance-issues/)的问题

### 3：追踪浏览器：查找性能问题的根本原因和常见模式

虽然性能问题和洞察为您提供了性能数据的高级视图，但[追踪浏览器](/product/explore/traces/)允许您按任何属性对您的所有性能数据进行切片和切块。这是查找性能回归或潜在优化的根本原因和常见模式的极其强大的工具。例如，您可以找到 `GET /users/:id` 端点上所有跨度的 p95 持续时间，或者找到由于特定中间件路由而变慢的所有页面。

最新版本的追踪浏览器还允许您快速计算基于跨度数据的指标，如 p50、p95 和 p99 持续时间，然后按任何属性对它们进行分组。这使您可以快速查看是否有特定的端点、版本、用户组等表现不佳。要更深入地了解如何使用新的追踪浏览器和跨度指标，请查看我们最近的[博客文章](https://blog.sentry.io/find-and-fix-performance-bottlenecks-with-sentrys-trace-explorer/)。

### 4：指标、警报和仪表板：构建性能数据的自定义视图

默认情况下，Sentry 自动跟踪每个追踪的一组[性能指标](/platforms/javascript/tracing/span-metrics/)。这些用于填充洞察仪表板，但您也可以使用它们来构建自定义[仪表板](/product/dashboards/)和[警报](/product/alerts/)。

选择加入追踪浏览器的[早期采用者](https://sentry.io/orgredirect/settings/:orgslug/)计划还可以让您访问新的[跨度指标](/product/explore/new-trace-explorer/#span-metrics/)功能，该功能允许您根据跨度数据计算自定义指标。这对于构建由真实性能追踪数据支持的仪表板和警报非常有用，因此您可以快速调试性能问题。

## 如何设置性能监控

对于基本的前端或后端性能监控，只需在相关 SDK 中打开追踪即可。这将自动追踪每个网络请求并为您提供一组基本的性能指标。但是，为了确保 Sentry 捕获准确和完整的性能数据，您需要设置[分布式追踪](/product/sentry-basics/distributed-tracing/)。NextJS、Laravel、Ruby on Rails 和其他一些框架对分布式追踪有很强的开箱即用支持，但对于更复杂或多语言的应用程序，您需要手动配置此功能。在[此处](/platforms/javascript/tracing/distributed-tracing/)了解有关追踪传播的更多信息。

## 性能监控在底层如何工作

Sentry 不是像在 `onLoad`、`onUnload`、`onError` 等特定时间间隔用特定指标检测页面加载，而是努力自动检测您依赖的库，为 Web 指标、文件 I/O、网络请求等提供详细的遥测。这使您可以看到每个请求的性能影响以及该请求期间发生的任何错误。

这与其他性能监控工具有很大不同，后者仅捕获具有更粗粒度的*预定义*指标。Sentry 中的所有整体指标都由原始追踪数据支持，因此您可以深入了解任何特定请求或操作以获得其性能的详细视图。

这些性能指标由 Sentry SDK 作为每个[跨度](/concepts/key-terms/tracing/#whats-a-span)的一部分发送，跨度是[追踪](/concepts/key-terms/tracing/)中的最小工作单位。追踪在 Sentry 的后端组装，然后推断以填充我们用于监控性能的所有工具。

这与其他性能监控工具有很大不同，后者仅在应用程序的特定点捕获指标。相反，您的所有整体指标都由原始追踪数据支持，因此您可以深入了解任何特定请求或操作以获得其性能的详细视图。

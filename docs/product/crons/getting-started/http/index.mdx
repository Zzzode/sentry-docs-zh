Sentry Crons 允许您监控任何计划、重复作业的正常运行时间和性能。实施后，它将允许您获取警报和指标，帮助您解决错误、检测超时并防止服务中断。

## 要求

要开始监控您的重复、计划作业：

1. 在 Sentry 中[创建新的监控器](https://sentry.io/issues/alerts/new/crons/)。
2. 为您的作业配置检查点或心跳。

您可以选择跳过第一步，通过检查点[创建或更新（upsert）监控器](#creating-or-updating-a-monitor-through-a-check-in-optional)。请参见下文。

## 检查点（推荐）

检查点监控允许您通过完成两个检查点来跟踪作业的进度：一个在作业开始时，另一个在作业结束时。这个两步过程允许 Sentry 在您的作业未按预期启动（错过）或超过其最大运行时间（失败）时通知您。


```bash {tabTitle: cURL}
SENTRY_INGEST="https://___ORG_INGEST_DOMAIN___"
SENTRY_CRONS="${SENTRY_INGEST}/api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/"

# 🟡 通知 Sentry 您的作业正在运行：
curl "${SENTRY_CRONS}?status=in_progress"

# 在此处执行您的计划任务...

# 🟢 通知 Sentry 您的作业已成功完成：
curl "${SENTRY_CRONS}?status=ok"
```

```http {tabTitle: HTTP}
GET /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/?status=in_progress HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___

GET /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/?status=ok HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___
```

如果您的作业执行失败：


```bash {tabTitle: cURL}
# 🔴 通知 Sentry 您的作业已失败：
curl "${SENTRY_CRONS}?status=error"
```

```http {tabTitle: HTTP}
GET /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/?status=error HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___
```

<Alert>

如果您期望您的计划作业重叠，请阅读下面的[重叠作业](#overlapping-jobs-optional)。

</Alert>

### 指定监控器环境（可选）

向监控器发送检查点时，您可以指定检查点的
`environment`。这允许您跨多个
环境监控单个计划。

如果您没有为检查点指定环境，则默认为 `production`。

<Alert>

监控器环境仍在开发中。目前，在为特定环境
发生检查点后，您必须继续按计划
发送检查点或删除监控器环境；否则，它将被标记为错过。

</Alert>


```bash {tabTitle: cURL}
SENTRY_INGEST="https://___ORG_INGEST_DOMAIN___"
SENTRY_CRONS="${SENTRY_INGEST}/api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/"

# 🟡 通知 Sentry 您的作业正在开发环境中运行：
curl "${SENTRY_CRONS}?environment=dev&status=in_progress"

# 在此处执行您的计划任务...

# 🟢 通知 Sentry 您的开发环境作业已成功完成：
curl "${SENTRY_CRONS}?environment=dev&status=ok"
```

### 通过检查点创建或更新监控器（可选）

Sentry 允许通过检查点有效负载自动创建或更新监控器（upsert）。如果您有许多计划任务或需要动态创建它们，这可能很有用。


```bash {tabTitle: cURL}
SENTRY_INGEST="https://___ORG_INGEST_DOMAIN___"
SENTRY_CRONS="${SENTRY_INGEST}/api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/"

# 🟡 通知 Sentry 您的作业正在运行：
curl -X POST "${SENTRY_CRONS}" \
    --header 'Content-Type: application/json' \
    --data-raw '{"monitor_config": {"schedule": {"type": "crontab", "value": "0 * * * *"}}, "status": "in_progress"}'
```

```http {tabTitle: HTTP}
POST /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/ HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___
Content-Type: application/json

{
    "monitor_config": {
        "schedule": {"type": "crontab", "value": "0 * * * *"},
        "checkin_margin": 1,
        "max_runtime": 20,
        "timezone": "America/Los_Angeles"
        "failure_issue_threshold": 1,
        "recovery_threshold": 1,
    },
    "status": "in_progress"
}
```

监控器 `monitor_config` 参数：

`schedule_type`：

监控器的计划表示，可以是 `crontab` 或 `interval`。

`schedule`：

作业的计划：

这是一个指定 `schedule_type` 为 `crontab` 或 `interval` 的对象。结构将根据类型而变化：

```json
{"type": "crontab", "value": "0 * * * *"}
{"type": "interval", "value": "2", "unit": "hour"}
```

`checkin_margin`：

Sentry 应等待您的检查点的时间量（以分钟为单位），然后才被视为错过（"宽限期"）。可选。

<Alert>

我们建议您将检查点边距设置为小于或等于您的间隔。

</Alert>

`max_runtime`：

您的作业被允许运行的时间量（以分钟为单位），然后才被视为失败。可选。

`failure_issue_threshold`：

：连续失败的检查点导致创建问题之前所需的次数。可选。

`recovery_threshold`：

：连续 OK 检查点导致解决问题之前所需的次数。可选。

`timezone`

您的作业运行的 `tz` 位置。这通常是您服务器的时区，（例如 `America/Los_Angeles`）。请参见[tz 数据库时区列表](https://en.wikipedia.org/wiki/List_of_tz_database_time_zones)。可选。

<Alert level="warning">

为非重复的 crontab 计划提供时区很重要，例如 `0 17 * * *`（每天下午 5 点）。

</Alert>

### 重叠作业（可选）

在前一个作业执行完成之前开始的作业执行称为重叠作业。如果您的作业运行时间超过作业的计划间隔，就会发生这种情况。

一个简单的例子是，如果您有一个每分钟运行一次（1）的作业，但每次执行需要五分钟（5）。

如果发生这种情况，您必须为每次请求提供稳定的检查点 ID，以防止意外后果。

使用示例：


```bash {tabTitle: cURL}
CHECK_IN_ID="$(uuidgen)"

SENTRY_INGEST="https://___ORG_INGEST_DOMAIN___"
SENTRY_CRONS="${SENTRY_INGEST}/api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/"

# 🟡 使用检查点 ID 通知 Sentry 您的作业正在运行：
curl "${SENTRY_CRONS}?check_in_id=${CHECK_IN_ID}&status=in_progress"

# 在此处执行您的计划任务...

# 🟢 使用检查点 ID 通知 Sentry 您的作业已成功完成：
curl "${SENTRY_CRONS}?check_in_id=${CHECK_IN_ID}&status=ok"
```

## 心跳

心跳监控通过一个检查点通知 Sentry 作业的状态。此设置仅在您的作业未按预期启动（错过）时通知您。如果您需要跟踪作业以查看它是否超过其最大运行时间（失败），请改用检查点。


```bash {tabTitle: cURL}
SENTRY_INGEST="https://___ORG_INGEST_DOMAIN___"
SENTRY_CRONS="${SENTRY_INGEST}/api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/"

# 🟢 通知 Sentry 您的作业已成功完成：
curl "${SENTRY_CRONS}?status=ok"
```

```http {tabTitle: HTTP}
GET /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/?status=ok HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___
```

如果您的作业执行失败：


```bash {tabTitle: cURL}
# 🔴 通知 Sentry 您的作业已失败：
curl "${SENTRY_CRONS}?status=error"
```

```http {tabTitle: HTTP}
GET /api/___PROJECT_ID___/cron/<monitor_slug>/___PUBLIC_KEY___/?status=error HTTP/1.1
Host: ___ORG_INGEST_DOMAIN___
```

## 警报

当您的重复作业未能检查点（错过）、运行超出其配置的最大运行时间（失败）或手动报告失败时，Sentry 将创建一个带有指向您的监控器标签的错误事件。

要接收有关这些事件的警报：

1. 在侧边栏中导航到**警报**。
2. 创建新警报并在"错误"下选择"问题"作为警报类型。
3. 配置您的警报并定义要使用的过滤器匹配：`事件的标签匹配 {键} {匹配} {值}`。

示例：`事件的标签匹配 monitor.slug 等于 my-monitor-slug-here`

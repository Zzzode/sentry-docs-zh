Sentry仅在[调试信息文件](/platforms/native/data-management/debug-files/)上传后才使用它们。_重新处理_允许您将调试信息文件应用于已经发生的错误事件。

<Include name="only-error-issues-note.mdx" />

在查看问题时，您可以找到以下操作：

!["重新处理事件"操作在问题详情页面的高级操作中](./img/reprocessing-action.png)

对话框需要两个答案：

- 要重新处理多少个事件：留空时，所有事件都会被重新处理
  并且如果堆栈跟踪发生显著变化，可能会移动到其他（新的或现有的）问题。
  如果您提供一个数字，例如20，那么最近的20个事件将被
  重新处理，任何剩余的事件将保持不变。

- 对剩余事件做什么：如果您选择只重新处理，例如100个事件中的20个，
  您可以选择对剩余的80个事件做什么。
  默认是将它们保留在原处（您正在查看的问题）。
  或者，您可以选择删除它们。

按下"重新处理事件"后，**问题被锁定并从您的问题流中隐藏，而重新处理正在进行。**当数据在问题之间迁移时，[sentry.io](https://sentry.io)不允许任何操作，例如解决或分配问题，以确保数据一致性并避免数据丢失。

请注意，您仍然可以通过搜索`is:reprocessing`（删除所有其他搜索词）手动访问问题，如果需要的话。

## 注意事项

开始重新处理的对话框指出了一些注意事项，您
在开始重新处理之前应该了解并理解这些注意事项。

- **数据故障。**在重新处理期间，您可能会观察到整个产品中的临时数据
  不一致。您可能会观察到事件同时属于
  两个问题，并在Discover查询中出现两次。

  这些不一致在重新处理完成后消失。然而，
  根据您的平台和涉及的事件数量，重新处理可能需要
  一段时间。

- **需要启用附件存储。**如果您的事件来自
  minidumps或unreal崩溃报告，您必须启用[附件
  存储](/platforms/native/enriching-events/attachments/#crash-reports-and-privacy)。
  **如果原始minidump在Sentry中不再存在，发送
  事件通过重新处理将导致它根本没有堆栈跟踪。**

- **配额适用。**您选择重新处理的每个事件都会
  针对您计划的配额计算第二次。速率限制和峰值保护不适用。

- **调试文件不会立即应用。**当您上传调试信息
  文件时，无论您是上传到自己的符号服务器还是Sentry，调试信息文件都是
  仅在内部缓存过期后一小时应用。请等待
  上传DIF后至少一小时再重新处理。

- **警报不适用，但有注意事项。**问题警报不会为
  重新处理的事件触发，新事件不受数据转发的影响。

  指标警报通常也不会触发。然而，它们受
  上面列出的数据不一致的影响。如果您创建了
  具有24小时测量窗口的指标警报，并且您重新处理的事件发生在
  过去24小时内，在事件重新处理期间将暂时被计算两次。
  这可能会触发指标警报。

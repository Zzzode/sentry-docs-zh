移动应用程序中的主（或UI）线程负责处理所有用户交互，需要能够实时响应手势和点击。如果长时间运行的操作阻塞了主线程，应用程序会变得无响应，影响用户体验质量。

将压缩的JPEG或PNG图像解码为位图的过程（特别是对于大图像），是可能影响应用程序响应性的长时间运行操作的一个示例。如果在主线程上检测到长时间运行的图像解码操作，Sentry会将其标记为问题。

## 检测标准

必须启用[性能分析](/product/explore/profiling/)，Sentry才能检测主线程解码问题。Sentry会查找在主线程上运行至少40ms的与图像解码相关的函数。此阈值确保函数在性能分析器收集的足够数量的样本（~10ms每个样本x 4 = ~40ms总计）中出现，然后才会被检测为问题。

## 函数证据

要查找有关主线程图像解码问题的额外信息，请转到其**问题详情**页面并向下滚动到"函数证据"部分，该部分显示以下内容：

- **事务名称：** 检测到问题的事务名称
- **可疑函数：** 触发问题检测的函数（在这种情况下，是图像解码函数）
- **持续时间：** 函数执行所需的时间以及性能分析器收集的包含该函数的连续样本数量

## 示例

### iOS

以下代码通过设置`UIImageView`的`image`属性来渲染图像。当`UIImage.init(data:)`被调用时，图像不会被解码，但当图像在屏幕上渲染时，这发生在主线程上。

```swift
func showImage(data: Data) {
	imageView.image = UIImage(data: data)
}
```

性能可以通过使用`[UIImage.preparingForDisplay()](https://developer.apple.com/documentation/uikit/uiimage/3750834-preparingfordisplay)`在后台队列上准备图像以供显示，然后在主队列上将图像分配给图像视图来改善：

```swift
func showImage(data: Data) {
	DispatchQueue.global(qos: .userInitiated).async {
		let image = UIImage(data: data).preparingForDisplay()
		DispatchQueue.main.async {
			imageView.image = image
		}
	}
}
```

### Android

以下代码在UI线程上调用`BitmapFactory.decodeByteArray`来解码图像并在`ImageView`中渲染它：

```kotlin
fun showImage(data: ByteArray) {
	imageView.setImageBitmap(BitmapFactory.decodeByteArray(data, 0, data.size))
}
```

性能可以通过将图像解码移动到协程来改善：

```kotlin
fun showImage(data: ByteArray) {
	lifecycleScope.launch(Dispatchers.IO) {
	  val bitmap = BitmapFactory.decodeByteArray(data, 0, data.size)

    withContext(Dispatchers.Main) {
      imageView.setImageBitmap(bitmap)
    }
  }
}
```

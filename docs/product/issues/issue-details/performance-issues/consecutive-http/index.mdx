当通过并行化最少3个按顺序发生的连续HTTP调用可以节省至少2000ms时间时，会创建连续HTTP问题。

## 检测标准

<Alert>

此问题目前仅适用于后端和移动端SDK。

</Alert>

此性能问题的检测器会查找一组顺序的、不重叠的HTTP跨度。它有意忽略非HTTP跨度，因为浏览器通常执行其他异步操作，例如在HTTP跨度之间加载资源。

一旦找到此序列，每个HTTP跨度必须满足以下条件：

- HTTP方法必须是GET、POST、DELETE、PUT或PATCH
- HTTP URL不能包含`\_next/static/`、`\_next/data/`或`googleapis.com`

一旦找到这些跨度，还必须满足以下条件：

- 并行化节省的最小时间必须超过`2000ms`
- 每个HTTP跨度的持续时间必须超过`500ms`的阈值
- 每个HTTP跨度之间的时间不得超过`500ms`的阈值
- 顺序HTTP跨度的数量必须超过`3`的阈值

如果Sentry没有在您期望的地方检测到连续HTTP问题，可能是因为事务未满足上述任一标准。

您可以在**项目设置 > 性能**中配置连续HTTP问题的检测器阈值：

![连续HTTP检测器阈值设置](./img/consecutive-http-threshold-settings.png)

## 跨度证据

您可以通过查看"跨度证据"部分中的两个主要方面来找到有关连续HTTP问题的额外信息：

- **事务名称**
- **顺序HTTP跨度**

![连续HTTP跨度证据](./img/span-evidence.png)

您可以通过转到Sentry中的**问题**页面，选择您的项目，选择您要检查的连续HTTP查询错误，然后在"详情"选项卡中向下滚动到"跨度证据"部分来查看跨度证据。

## 修复连续HTTP问题

连续HTTP问题的原因各不相同，但以下是一些帮助您解决它们的建议：

- 如果您的HTTP跨度是独立的，请考虑并行化它们，使它们不再顺序执行。
- 如果您可以控制后端，请考虑为此特定用例优化您的端点。您可以调查相应的后端事务以帮助识别潜在的改进。

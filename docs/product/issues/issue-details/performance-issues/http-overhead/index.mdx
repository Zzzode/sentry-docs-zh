当浏览器排队的一组重叠HTTP跨度具有相同主机且排队时间持续超过500ms时，会创建HTTP/1.1开销问题。

## 检测标准

<Alert>

此问题目前仅适用于浏览器SDK。

</Alert>

此性能问题的检测器会查找一组重叠的HTTP/1.1跨度，这些跨度由于请求本身被延迟而变慢。由于浏览器连接限制，这对HTTP/1.1来说是个问题。

一旦找到这些跨度，还必须满足以下条件：

- 集合中最慢跨度达到的最大队列或'等待'时间必须超过`500ms`的阈值
- 必须有`6`个或更多重叠的HTTP跨度
- 每个HTTP跨度必须使用`HTTP/1.1`
- 队列时间必须对每个新的HTTP跨度增加
- 所有跨度必须共享一个公共主机（HTTP连接来自同一源或同一域名）

所有HTTP跨度都会经历一些等待时间，并将包含在检测中，但只有超过`100ms`的HTTP跨度会被标记为'违规'，并会显示在跨度证据中。

如果Sentry没有在您期望的地方检测到HTTP开销问题，可能是因为事务未满足上述任一标准。您可以通过转到**设置**页面，选择"项目"，选择您感兴趣的项目，选择"性能"，然后向下滚动到"性能问题 - 检测器阈值设置"来进行任何更新，从而配置Sentry中的HTTP开销问题的检测器阈值。

## 跨度证据

您可以通过查看"跨度证据"部分中的两个主要方面来找到有关连续HTTP问题的额外信息：

- **事务名称**
- **最大队列时间**

![HTTP开销跨度证据](./img/http-overhead-spans.png)

您可以通过转到Sentry中的**问题**页面，选择您的项目，选择您要检查的HTTP开销错误，然后在"详情"选项卡中向下滚动到"跨度证据"部分来查看跨度证据。

## 如何修复HTTP开销

HTTP开销问题可以通过升级到允许HTTP/2或更高版本连接到您的后端来解决。这将涉及您的前端服务器（如果您有负载均衡器），或您的服务器（如果可以直接访问）。由于这可能并不总是可能的，其他选项是执行域分片，或减少到后端的同时请求数量，但这些解决方案更手动且更难长期维护。

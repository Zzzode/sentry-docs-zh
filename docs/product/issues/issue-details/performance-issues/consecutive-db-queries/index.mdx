连续数据库查询是一系列数据库跨度，其中一个或多个被识别为可并行化，换句话说，是可以移动到序列开头的跨度。这通常发生在数据库查询不对数据执行过滤时，例如没有WHERE子句的查询。

## 检测标准

此性能问题的检测器会查找一组顺序的、不重叠的数据库跨度。在这组跨度中，它会继续查找可并行化的查询，当查询满足以下条件时会发生这种情况：

- 不包含`WHERE`子句
- 是`SELECT`查询
- 查询中没有参数
- 不是一组顺序跨度中的第一个查询

一旦找到这些跨度，还必须满足以下条件：

- 并行化节省的最大时间必须超过100ms阈值
- 节省的最大时间与顺序跨度持续时间之间的比率必须超过0.1阈值
- 每个可并行化跨度的总持续时间必须超过30ms阈值

如果Sentry没有在您期望的地方检测到连续数据库问题，可能是因为事务未满足上述任一标准。

您可以在**项目设置 > 性能**中配置连续数据库查询问题的检测器阈值：

![连续数据库查询检测器阈值设置](./img/consecutive-db-detector-settings.png)

## 跨度证据

您可以通过"跨度证据"部分中的四个主要方面来识别连续数据库查询问题：

- 事务名称
- 起始跨度 - 一组连续跨度中的第一个跨度
- 可并行化跨度 - 可以并行化的跨度
- 持续时间影响 - 由于跨度未并行运行而添加到总事务时间的分数

![连续数据库查询跨度证据](./img/span-evidence.png)

通过转到Sentry中的**问题**页面，选择您的项目，点击您要检查的连续数据库查询错误，然后在"详情"选项卡中向下滚动到"跨度证据"部分来查看它。

## 示例

在从数据库计数和检索所有用户时，请考虑使用此Node.js代码

```javascript
const result = await db.query("SELECT * FROM USERS");
const count = await db.query("SELECT COUNT(*) FROM USERS");
processMyData({ result, count });
```

这会导致连续数据库性能问题。您会注意到这些查询在跨度树中按顺序发生，但它们不是相互依赖的，因为一个的结果不会影响另一个。因此，它们可以并行执行。

![连续数据库查询示例](./img/consecutive-example.png)

要修复此性能问题，您可以使用Promise.all()：

```javascript
const [result, count] = await Promise.all([
  db.query("SELECT * FROM USERS"),
  db.query("SELECT COUNT(*) FROM USERS"),
]);
processMyData({ result, count });
```

这使查询并行执行，在这种情况下，事务持续时间减少了近一半！

![连续数据库查询已解决](./img/parallel-example.png)

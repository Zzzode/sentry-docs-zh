函数回归问题捕获了一类通用问题，其中函数的持续时间随时间增加。这可能导致应用程序性能下降。Sentry尝试开箱即用地监控您的一些最常见函数，并在函数回归问题类型下报告任何可能的回归。

## 检测标准

Sentry只能在启用了[性能分析](/product/explore/profiling)的项目上检测函数回归问题。Sentry会定期检查项目中最常见应用程序函数的第95百分位函数持续时间。当检测到函数持续时间的p95值显著增加并已持续一段时间时，会创建回归问题。

## 图表

图表显示了在识别回归之前最多14天和之后最多14天的窗口。在这里，您可以看到p95函数持续时间以及函数吞吐量。

回归的常见原因是函数吞吐量的增加。例如，当包含互斥锁的代码路径中的吞吐量增加时，这可能导致锁争用。

## 函数证据

要查找有关回归函数的额外信息，请转到"函数证据"部分，该部分显示以下内容：

- **函数名称：** 已回归的函数名称
- **包名称：** 已回归函数的包
- **文件名称：** 已回归函数的文件名
- **回归日期：** 回归开始的日期和时间

## 受影响的事务

"受影响的事务"部分显示了此回归出现的事务列表。在每个事务旁边，您可以找到以下内容：

- 回归前回归函数的p95函数持续时间
- 回归后回归函数的p95函数持续时间
- 回归前后p95函数持续时间的百分比变化

前后值还链接到您可以检查以识别回归根本原因的示例性能分析。

## 示例性能分析

"示例性能分析"部分显示了示例性能分析列表，您可以在其中找到回归前后的回归函数。

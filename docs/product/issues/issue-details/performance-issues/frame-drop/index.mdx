移动应用程序中的主（或UI）线程负责处理所有用户交互，需要能够实时响应手势和点击。如果长时间运行的操作阻塞了主线程，应用程序会变得无响应，影响用户体验质量。

我们可以检测主线程停顿的某些特定原因，如解码[图像](/product/issues/issue-details/performance-issues/image-decoding-main-thread/)或[JSON](/product/issues/issue-details/performance-issues/json-decoding-main-thread/)，或[使用正则表达式搜索字符串](/product/issues/issue-details/performance-issues/regex-main-thread/)，但还有许多其他可能导致停顿的原因。如果主线程停顿导致您的应用程序丢失UI帧，但未匹配我们的特定检测器之一，Sentry会在通用帧丢失问题类型下报告它。

## 检测标准

必须启用[性能分析](/product/explore/profiling/)，Sentry才能检测帧丢失问题。设置完成后，Sentry会查找记录冻结UI帧的性能分析，然后搜索最耗时的应用程序函数调用，该调用延迟了显示链接的下一个vsync。在一系列没有自身时间的函数调用中，将选择最深的调用。

<Alert>

Cocoa的最低支持版本为`8.12.0`。

</Alert>

## 示例

### iOS

以下代码在主线程上执行长时间运行的`while`循环：

```objc
// 给定一个数字字符串数组...

NSMutableOrderedSet<NSString *> *sortedEvenNumbers = [NSMutableOrderedSet<NSString *> orderedSet];
[numbers enumerateObjectsUsingBlock:^(NSString * _Nonnull obj, NSUInteger __unused idx, BOOL * _Nonnull __unused stop) {
    if (obj.integerValue % 2 == 0) {
        [sortedEvenNumbers addObject:obj];
    }
}];
```

性能可以通过将长时间计算移出主线程来改善。最简单的方法是将其调度到较低服务质量(QoS)队列（或`NSOperationQueue`）：

```objc {tabTitle:调度到后台队列}
// 给定一个数字字符串数组...

NSMutableOrderedSet<NSString *> *sortedEvenNumbers = [NSMutableOrderedSet<NSString *> orderedSet];
dispatch_async(dispatch_get_global_queue(QOS_CLASS_UTILITY, 0), ^{
    [numbers enumerateObjectsUsingBlock:^(NSString * _Nonnull obj, NSUInteger __unused idx, BOOL * _Nonnull __unused stop) {
        if (obj.integerValue % 2 == 0) {
            [sortedEvenNumbers addObject:obj];
        }
    }];
});
```

另一个需要考虑的途径是并行化循环迭代。有几种方法可以做到这一点：

1. 如果您正在遍历Foundation集合，您可能已经在使用`enumerateObjects`或`enumerateKeysAndObjects`。将其更改为`-[NSArray|NSSet|NSOrderedSet enumerateObjectsWithOptions:usingBlock:]`或`-[NSDictionary enumerateKeysAndObjectsWithOptions:usingBlock:]`，选项为`NSEnumerationConcurrent`。请注意，从块内部修改集合将导致抛出异常。这大致相当于将循环体的每次迭代调度到并发GCD队列。

1. 使用`dispatch_apply`在并发队列上执行一般循环的迭代。

```objc {tabTitle:Foundation集合并发枚举}
// 给定一个数字字符串数组...

NSMutableOrderedSet<NSString *> *sortedEvenNumbers = [NSMutableOrderedSet<NSString *> orderedSet];
[numbers enumerateObjectsWithOptions:NSEnumerationConcurrent usingBlock:^(NSString * _Nonnull obj, NSUInteger __unused idx, BOOL * _Nonnull __unused stop) {
    if (obj.integerValue % 2 == 0) {
        [sortedEvenNumbers addObject:obj];
    }
}];
```

```objc {tabTitle:dispatch_apply}
// 给定一个数字字符串数组...

NSMutableOrderedSet<NSString *> *sortedEvenNumbers = [NSMutableOrderedSet<NSString *> orderedSet];
dispatch_apply(numberOfNumbers, dispatch_get_global_queue(QOS_CLASS_UTILITY, 0), ^(size_t iteration) {
    NSString *number = numbers[iteration];
    if (number.integerValue % 2 == 0) {
        [sortedEvenNumbers addObject:number];
    }
});
```

引入并发时需要记住几件事：

- 您可能需要`@synchronize`关键部分，使用信号量，或调度回串行队列（或主队列用于UI工作）。
- 您可能无法并行化迭代相互依赖或顺序重要的循环。
- 并行化对于小集合可能效率较低，因为[线程生成有其自身的成本](https://developer.apple.com/library/archive/documentation/Cocoa/Conceptual/Multithreading/AboutThreads/AboutThreads.html#//apple_ref/doc/uid/10000057i-CH6-SW20)。因此，始终先测量！
- `enumerateObjects...`和`dispatch_apply`都是同步的，在所有迭代完成之前不会返回。异步调度它们的调用，使其脱离主队列以避免等待。

### Android

以下函数执行长时间运行的`for`循环来对列表进行排序：

```kotlin
// 对数字列表进行排序，偶数在奇数之前
fun sortEvenOdd(numbers: List<Int>): List<Int> {
    val newList = ArrayList<Int>()
    for (n in numbers) {
        if (n % 2 == 0) {
            val i = newList.indexOfFirst { it % 2 == 1 }
            newList.add(i.takeIf { it != -1 } ?: 0, n)
        } else {
            val i = newList.indexOfLast { it % 2 == 1 }
            newList.add(i.takeIf { it != -1 } ?: newList.size, n)
        }
    }
    return newList
}
```

性能可以通过将长时间计算移出主线程到协程来改善：

```kotlin
// 对数字列表进行排序，偶数在奇数之前，然后调用onSorted并将排序后的列表作为参数
fun sortEvenOdd(numbers: List<Int>, onSorted: (sorted: List<Int>) -> Unit): List<Int> {
    lifecycleScope.launch(Dispatchers.Default) {
    val newList = ArrayList<Int>()
        for (n in numbers) {
            if (n % 2 == 0) {
            if (n % 2 == 0) {
                val i = newList.indexOfFirst { it % 2 == 1 }
                newList.add(i.takeIf { it != -1 } ?: 0, n)
            } else {
                val i = newList.indexOfLast { it % 2 == 1 }
                newList.add(i.takeIf { it != -1 } ?: newList.size, n)
            }
        }
        withContext(Dispatchers.Main) {
            // 继续在主线程上计算
            onSorted(newList)
        }
    }
}
```

没有通用的方法来修复自定义算法的性能，因为每个算法都不同。

在我们上面的示例中，对于列表中的每个项目，我们分析每个数字，执行搜索以找到放置元素的正确索引，最后将其插入到列表中。

通过为偶数和奇数使用两个单独的列表，我们可以避免搜索步骤，因为正确的索引始终是列表的末尾。
此外，由于ArrayList的性质，在列表中间插入元素比在末尾插入要慢。

考虑到这些因素，可以使此函数更快：

```kotlin
// 对数字列表进行排序，偶数在奇数之前，然后调用onSorted并将排序后的列表作为参数
fun sortEvenOddFaster(numbers: List<Int>, onSorted: (sorted: List<Int>) -> Unit): List<Int> {
    lifecycleScope.launch(Dispatchers.Default) {
        val evenList = ArrayList<Int>(numbers.size)
        val oddList = ArrayList<Int>(numbers.size)
        for (n in numbers) {
            if (n % 2 == 0) {
                evenList.add(n)
            } else {
                oddList.add(n)
            }
        }
        evenList.addAll(oddList)
        withContext(Dispatchers.Main) {
            // 继续在主线程上计算
            onSorted(evenList)
        }
    }
}
```

要记住几个注意事项来改善算法的性能：

- 您可以通过使用线程（或Kotlin协程）来并行化工作。
- 探索并为您的特定范围找到最佳数据结构。
- 某些结构可以被缓存并在整个函数中重用，减少内存使用。
- 评估解决问题的不同方法。

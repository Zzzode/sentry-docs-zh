_N+1 API调用_是一种性能问题，当应用程序进行许多同时的网络请求来获取给定类型的资源时发生。例如，当应用程序为列表中的每个项目进行一个网络请求，而不是在单个组合网络调用中一次性获取所有信息时。每次对服务器的调用都会产生性能开销，如果调用数量很高，可能会导致显著的性能问题。当渲染列表或项目网格时，每个项目都需要从服务器获取额外信息时，通常会出现此问题。

## 受影响的平台

Sentry在前端和移动应用程序中检测N+1 API调用。检测器与语言和平台无关。您可以在React、Vue、Angular、React Native、Android等环境中看到此问题。目前不支持后端环境。

## 检测标准

N+1 API调用检测器会查找对同一URL的同时网络调用组。它使用以下检测标准：

- 仅适用于`GET`请求
- 必须同时发生至少`10`个网络调用
- 涉及跨度的总持续时间必须超过`300ms`
- 网络调用必须在`5ms`内彼此发生
- 调用不能是GraphQL端点或`_next/data`
- 调用不能是获取静态资源（如`.png`文件）
- 事务操作必须是`"navigation"`、`"pageload"`、`"ui.load"`或`"ui.action"`
- 跨度操作必须是`"http.client"`

您可以在**项目设置 > 性能**中配置N+1 API调用问题的检测器阈值：

## 跨度证据

您可以通过查看"跨度证据"部分中的三个主要方面来识别N+1 API调用问题的根本原因：

- 事务名称
- 重复跨度 - 这是接收重复网络调用的URL
- 参数 - 这些是网络调用之间变化的查询参数

![N+1 API调用跨度证据](./img/n-plus-one-api-calls-evidence.png)

通过转到Sentry中的**问题**页面，点击您要检查的N+1 API调用错误，然后在"详情"选项卡中向下滚动到"跨度证据"部分来查看它。

## 指纹识别

Sentry根据重复网络调用的URL计算问题指纹。Sentry通过剥离整数、UUID、SHA1哈希和MD5哈希来参数化URL。如果您发现Sentry未能正确分组N+1 API调用问题，您可以使用<PlatformLink to="/configuration/options#before-send-transaction">before-send-transaction</PlatformLink>钩子（如果您的平台支持）来参数化事务中的`http.client`跨度。

## 示例

考虑一个用React编写的图书推荐网站。主页生成最受欢迎图书的列表，并为列表中的每本书渲染一个`<Book />`组件。每个`<Book />`组件都会进行`fetch`调用来从API获取图书信息。

```jsx
const Book({ id }) => {
  { book, isLoading } = useFetch(`/books?id=${id}`);

  if (isLoading) {
    return <span>LOADING...</span>
  }

  return (
    <div>
      <span>{book.title}</span>
    </div>
  );
};
```

如果列表中有100本书，页面将同时进行100个网络调用。数字100是"N+1 API调用"中的"N"。此效果在跨度瀑布图中可见，显示了所有同时的网络请求：

![示例应用程序中的N+1 API调用](./img/n-plus-one-api-calls-in-book-application.png)

## 修复N+1 API调用问题

此问题类型的修复取决于应用程序上下文。在某些情况下，此行为是故意的甚至是期望的。由您决定适当的修复是什么（如果有的话）。以下是一些常见方法：

- 提前在一个网络调用中获取所有信息。在上面的示例中，您可以更新`/books`端点以接受ID列表（例如，`/books?id=78&id=89&id=120`）。
- 延迟加载信息。在图书示例中，您可以更改`<Book>`组件，仅在组件在屏幕上可见时才获取信息。
- 缓存信息。如果您的页面进行可能是重复的并发调用，您可以通过添加缓存或调用去重来提高性能。

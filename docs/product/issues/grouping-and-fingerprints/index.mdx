![问题仪表板](./img/issues-dashboard.png)

有没有想过为什么某些事件会分组为单个问题，而其他事件则分组为单独的问题？

让我们首先了解错误和事务如何分组为一个问题。在上面的示例中，问题`testTypeIssue14`有718个事件，Sentry将它们分组为一个问题，因为它们共享相同的_指纹_。

**什么是指纹？**
指纹是一种唯一标识事件的方式。Sentry默认使用内置分组算法为您设置指纹，该算法基于堆栈跟踪、异常类型和消息等信息。具有相同指纹的事件被分组在一起。

**我需要做任何事情吗？** 不需要。Sentry根据事件类型和事件中的其他数据自动生成指纹。如果事件具有相同的根本原因，它们将分组为一个问题。如果您想更改事件的分组方式，请参见下面的更改事件分组方式。

**如何查看指纹？** 打开一个问题，点击JSON链接，然后找到_fingerprint_属性。如果使用了默认分组，您会看到`"{{ default }}"`写在那里。如果使用了不同的分组，您会看到指纹值本身。

## 更改事件的分组方式

<Alert>

本节中描述的解决方案仅适用于[错误问题](/product/issues/issue-details/error-issues/)。其他类别的问题（如[性能问题](/product/issues/issue-details/performance-issues/)）不支持此功能。

</Alert>

有四种不同的方法可以更改事件如何分组为问题。

**合并相似问题**：这允许将已存在的问题合并在一起。无需任何设置或配置即可执行此操作。

**指纹规则**：这允许您配置规则，直接设置未来事件的指纹。

**堆栈跟踪规则**：这允许您配置规则，在将分组规则应用于未来事件时使用哪些堆栈跟踪帧。

**SDK端指纹**：这允许您在事件发送到Sentry之前在事件上直接设置指纹。

请注意，配置指纹和堆栈跟踪规则或使用SDK端指纹都不会影响现有事件或问题，只会更改未来事件的分组方式。

请参见下文，了解每种方法的更多详细信息。

### 合并相似问题

这允许将两个或更多已存在的问题合并，这样任何本来会落入其中一个问题的未来事件都将被添加到合并的问题中。

合并前
![合并前](./img/before-merge-1.png)

合并后，我们看到`testTypeIssue7`和`testTypeIssue8`被合并在一起。
![合并后](./img/after-merge-1.png)

您也可以从单个问题内部执行此操作。点击相似问题选项卡
![相似问题](./img/similar-issues-1.png)

**警告：** 这不会影响未来问题的创建。换句话说，任何本来会创建新问题的未来事件仍将这样做，无论相似性如何。只有来自合并问题的事件才会包含在新的组合问题中。

如果您想取消合并问题，您可以转到**问题详情**页面并点击"合并问题"选项卡。您将看到已合并到该问题的指纹。选择适当的复选框并点击"取消合并"。

2. 在项目设置中设置匹配逻辑和相应的指纹。

### 堆栈跟踪规则

有时两个堆栈跟踪是相同的函数执行路径，但一个或多个帧对应于中间件、第三方库或框架本身的代码不同。Sentry将自动尝试检测并忽略此类帧，但这也是可配置的。请参见[堆栈跟踪规则](/concepts/data-management/event-grouping/stack-trace-rules/)。

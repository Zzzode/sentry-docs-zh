如果您启用了[洞察](/product/insights/)并且您的应用程序查询数据库，您可以在Sentry中查看查询的性能表现。

Sentry的查询监控帮助您调查查询的性能并获取更多信息来改进它们。

从[**查询**页面](#查询页面)开始，您可以获得查询性能的高级概览。从那里，您可以深入查看特定查询的[**查询摘要**页面](#查询摘要页面)，然后通过[样本列表](#样本列表)调查样本事件，以更好地了解其在特定端点中的性能上下文。

## 先决条件和限制

查询小组件和页面仅适用于启用了性能监控的后端项目。

### 推荐的SDK版本

查询监控在使用最新SDK版本时效果最佳。建议使用以下最低版本：

- [Python SDK v1.29.2](https://github.com/getsentry/sentry-python/releases/tag/1.29.2)
- [JavaScript SDKs v7.63.0](https://github.com/getsentry/sentry-javascript/releases/tag/7.63.0)
- [Laravel SDK v3.8.0](https://github.com/getsentry/sentry-laravel/releases/tag/3.8.0)
- [Cocoa SDK v8.11.0](https://github.com/getsentry/sentry-cocoa/releases/tag/8.11.0)
- [Java SDK v6.29.0](https://github.com/getsentry/sentry-java/releases/tag/6.29.0)
- [Ruby SDK v5.11.0](https://github.com/getsentry/sentry-ruby/releases/tag/5.11.0)

- [Android SDK v6.30.0](https://github.com/getsentry/sentry-java/releases/tag/6.30.0)

### Span资格

Sentry尝试为所有类SQL方言以及MongoDB提取指标。目前不支持其他NoSQL数据库如Elasticsearch、图数据库如Neo4j以及其他非SQL数据库系统。

如果您正在使用<PlatformLink to="/tracing/instrumentation/automatic-instrumentation">自动检测</PlatformLink>，查询性能监控应该无需任何配置即可工作。如果您手动检测了Sentry，您需要确保您的span符合我们的标准以获得最佳体验：

- span的`op`字段设置为[符合条件的值](https://develop.sentry.dev/sdk/performance/span-operations/#database)。
- span的描述包含完整的参数化查询（例如`"SELECT * FROM users WHERE id = ?"`）。SQL支持的占位符值为`%s`、`?`、`:c0`（例如`:c0`、`:c1`）和`$1`（例如`$1`、`$2`）。MongoDB span描述应包含有效的JSON，占位符应为字符串`"?"`。
- `db.system` span数据值设置为[正确的标识符](https://github.com/open-telemetry/semantic-conventions/blob/main/docs/database/database-spans.md)（例如`"postgresql"`或`"mongodb"`）。

## 查询页面

**查询**页面为您提供了所选项目中应用程序查询性能的快速概览。您可以将此页面作为起点来调查潜在的问题查询，并深入了解各种查询如何影响应用程序的性能。

点击侧边栏"洞察"下的"查询"打开**查询**页面。或者，**性能**页面也会在"最耗时的查询"小组件中显示影响最大的查询。您可以从那里点击"查看全部"打开**查询**页面。

在页面顶部，每分钟查询数（吞吐量）和平均持续时间的摘要图表为您提供了数据库性能的高级洞察。如果您发现异常或想进一步调查某个时间范围，您可以直接在图表中点击并拖动来选择该时间范围以筛选该时间范围的数据。

下面的查询描述表格显示了查询列表、它们的数量（每分钟查询数）、平均持续时间以及应用程序运行该查询所花费的总时间（花费时间）。查询描述已参数化，因此您的查询可能看起来略有不同。

默认情况下，此表格按最多[花费时间](#什么是花费时间)排序。这意味着顶部的查询通常运行非常缓慢、非常频繁，或两者兼有。您可以将其更改为按每分钟查询数排序以查看运行最频繁的查询，或按平均持续时间排序以查看最慢的查询。

您还可以使用上方的下拉菜单筛选特定命令（如`SELECT`和`UPDATE`）和查询的表/集合的所有数据。

要查看更多详细信息，请点击表格中的查询以打开其**查询摘要**页面。

### 查询参数化

有时，Sentry会处理查询并简化它们以提高可读性。例如，对于SQL查询：

- 如果可能，从列选择中删除表名（例如，`SELECT "users"."name", "users"."id", "users.age" FROM users`变为`SELECT name, id, age FROM users`）
- 折叠长列表的`SELECT`参数（例如，`SELECT id, name, age, city, country, phone FROM users`变为`SELECT .. FROM users`）
- 折叠长列表的`VALUES`参数（例如，`INSERT INTO users (id, email, name, age) VALUES (%s %s %s %s)`变为`INSERT INTO users (..) VALUES (%s)`）
- 折叠`CASE / WHEN`语句

对于MongoDB查询：

- 叶值被替换为占位符`"?"`
- 对深度嵌套的查询对象应用最大深度

您可以通过悬停在截断的描述上来查看完整查询，或点击它以查看其**查询摘要**页面。

## 查询摘要页面

要打开查询的**查询摘要**页面，请点击**性能**页面中"最耗时的查询"小组件中的表格或**查询**页面中的表格中的查询。

在页面顶部，显示所选时间范围的每分钟查询数、平均持续时间和花费时间。这些指标下方显示完整查询，然后是每分钟查询数和平均持续时间的图表。如果可用，还会显示查询在代码中的来源。

在底部，您可以找到查询所在的端点列表，按应用程序在该span中花费的时间最多排序。

如果您想调查特定端点，请点击它以打开侧边栏显示一些样本事件。

## 样本列表

点击端点以打开其样本列表。此侧面板显示查询在特定端点中的摘要指标（QPM和平均持续时间）。

表格显示了样本事件列表、它们的span持续时间以及与平均值的持续时间差异。此表格按最长span持续时间排序。

Sentry会自动查找各种样本以帮助您调查性能问题。所选事件将覆盖整个所选时间范围，以及一系列持续时间（比平均值快、慢和接近）。您可以通过比较不同的样本事件来了解性能特征。

这些相同的事件也显示为上方平均持续时间图表中的三角形。

您可以通过点击底部的"尝试不同样本"按钮生成新的随机样本事件列表。

从样本列表中，您可以深入了解给定端点中给定查询的特定好、平均或差示例。点击事件ID以深入了解查询span在[跟踪视图页面](/concepts/key-terms/tracing/trace-view/)的span瀑布中的详细信息。

## 什么是花费时间？

默认情况下，查询按**花费时间**排序，这可以作为给定查询相对影响的代理。

每个数据库查询都需要一些时间来执行。当应用程序发送查询时，它会等待数据库接收查询、解析查询、执行查询并返回结果。在Sentry中，从发送查询到接收完整结果所花费的总时间称为查询的**持续时间**。查询的**花费时间**是给定时间段内其所有持续时间的总和。

例如，应用程序可能会每次用户调用`/users`端点时发送像`SELECT * FROM users LIMIT 10`这样的查询。查询可能需要100毫秒到200毫秒来运行，并且端点可能每分钟被调用10次到100次，具体取决于一天中的时间。查询的花费时间是给定时间段内发生的其所有持续时间的总和。换句话说，花费时间是平均持续时间和每分钟查询数的乘积。在高吞吐量应用程序中，查询的花费时间可能以周或月为单位。

花费时间是衡量查询相对影响的有用方式，与应用程序中的其他查询相比。例如，如果一个查询的花费时间是7天，另一个是2小时，花费时间更多的查询可能更值得修复。即使持续时间的小幅改进也会导致有意义的整体改进。花费时间也可能是出问题的强烈信号。突然开始占用更多时间的查询通常表示存在问题。

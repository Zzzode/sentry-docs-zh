Sentry 为您的组织需求提供了多种配置选项来创建正常运行时间告警，如下所述。

## 1. 环境

首先，指定此告警规则所属的<PlatformLink to="/configuration/environments/">环境</PlatformLink>。从此告警规则创建的任何[正常运行时间问题](/product/issues/issue-details/uptime-issues/)都将设置为您的指定环境。

"环境"下拉列表显示项目中可用的相同环境，不包括隐藏的环境。

## 2. 项目

指定与您的告警规则关联的项目。创建的任何[正常运行时间问题](/product/issues/issue-details/uptime-issues/)将出现在此项目下。

## 3. 请求配置

![正常运行时间告警请求配置](./img/uptime-alert-request-config.png)

通过设置以下选项来配置 Sentry 如何执行 HTTP 正常运行时间检查：

- **间隔**：每次正常运行时间检查之间的时间。选项：`1 分钟`、`5 分钟`、`10 分钟`、`20 分钟`、`30 分钟` 和 `1 小时`。
- **超时**：Sentry 在将请求视为失败之前等待响应的最长时间（最多30秒）。
- **URL**：正常运行时间检查的目标 URL。
- **方法**：使用的 HTTP 方法（`GET`、`POST`、`HEAD`、`PUT`、`DELETE`、`PATCH` 或 `OPTIONS`）。
- **标头**：请求中包含的自定义标头。
- **正文**：请求有效负载，适用于 `POST`、`PUT` 和 `PATCH` 方法。
- **允许采样**：通过 Sentry SDK 启用请求的跨度采样。有关详细信息，请参阅[正常运行时间的分布式跟踪](/product/uptime-monitoring/uptime-tracing/)。

<Alert level="warning">

添加 HTTP 标头时，请小心不要包含敏感数据，例如 API 令牌或个人信息，以防止意外暴露或存储。

</Alert>

![正常运行时间告警预期请求](./img/uptime-alert-expected-check-request.png)

在请求配置下方，您将找到 Sentry 将向指定 URL 发送的预期请求示例，包括方法、标头和正文。Sentry 会自动添加 `User-Agent` 和 `Sentry-Trace` 标头。

其他注意事项：

- 如果目标 URL 需要，请包含 `Content-Type` 标头。例如，JSON 有效负载应具有 `Content-Type: application/json`。
- 选定的间隔会影响停机检测速度。Sentry 在三次连续失败后触发正常运行时间问题。例如，使用5分钟间隔，停机至少在首次失败后15分钟被检测到。了解更多关于[正常运行时间检查标准](/product/uptime-monitoring/#uptime-check-criteria)的信息。
- 如果指定的 URL 位于防火墙后面，请确保 Sentry 的正常运行时间机器人可以对其执行请求。了解更多关于[正常运行时间监控的防火墙配置](/product/uptime-monitoring/troubleshooting/#verify-firewall-configuration)的信息。
- Sentry 正常运行时间跟踪会自动将 sentry-trace 标头附加到传出请求以进行分布式跟踪。[了解更多](/product/uptime-monitoring/uptime-tracing/)。

## 4. 告警名称

为您的告警提供一个描述性名称，例如"着陆页"或"联系页面"。

## 5. 所有权

分配一个团队或团队成员来管理告警。如果未分配团队，任何用户都可以修改告警。从此告警规则创建的[正常运行时间问题](/product/issues/issue-details/uptime-issues/)将设置为指定的团队或团队成员。
